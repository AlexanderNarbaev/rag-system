# ADR-005: Версионированное индексирование документов

**Статус:** Принято
**Дата:** 2026-06-22

## Контекст

Корпоративные источники знаний (страницы Confluence, задачи Jira, merge requests GitLab) постоянно обновляются. Наивный подход «переиндексировать всё» тратит вычислительные ресурсы и выдаёт устаревшие результаты между запусками ETL. Пользователям нужна возможность запрашивать конкретные версии документов (например, «по состоянию на Q3 2025»), а система должна избегать одновременного показа конфликтующих версий одного документа. ETL-пайплайн запускается периодически на отдельной машине; прокси должен отражать обновления в течение минут.

Рассмотренные альтернативы: **полная переиндексация при каждом запуске ETL** (расточительно, высокая задержка), **сравнение по временным меткам** (хрупко, проблемы с расхождением часов), **API версий на уровне документов** (не все источники их предоставляют — API Jira и GitLab не имеют примитивов версионирования).

## Решение

**Реализовать контентно-адресуемое версионирование чанков через SHA-256 хеширование с паттерном горячего/холодного хранения LiveVectorLake и инкрементальным WAL-чейкпоинтингом.**

`ChunkVersionStore` (`etl/chunker/hash_versioning.py:52-275`) вычисляет SHA-256 хеши над содержимым чанка и ключевыми метаданными (`hash_versioning.py:30-49`). Хеш служит идентификатором точки Qdrant, делая каждый чанк уникально адресуемым. Инкрементальные обновления сравнивают хеши новых чанков с последним известным состоянием из WAL (`hash_versioning.py:115-161`), возвращая только добавленные и удалённые хеши.

`LiveVectorLake` (`etl/indexer/live_vector_lake.py:32-209`) стратифицирует хранение: горячий слой (Qdrant — актуальные чанки для быстрого поиска) и холодный слой (Parquet/Delta Lake — полная история версий с временными метками). Метод `sync_document()` (`live_vector_lake.py:110-140`) координирует оба слоя: новые чанки добавляются в Qdrant, удалённые чанки убираются, и все изменения добавляются в холодное хранилище.

WAL (`etl/indexer/wal_manager.py`) отслеживает чекпоинты каждого пайплайна (Confluence, Jira, GitLab, Indexing, Graph), обеспечивая возобновление после сбоя без потери данных. Оркестратор ETL (`etl/scheduler/run_etl.py:305`) обновляет WAL после завершения индексации.

Версионно-привязанный поиск поддерживается через фильтр Qdrant по payload-полю `version` (`proxy/app/retrieval.py:146-148`), доступный через параметр `rag_version` прокси.

## Последствия

**Положительные:** Инкрементальные обновления сокращают время ETL с часов до минут для больших репозиториев. Контентная адресуемость устраняет дубликаты — одинаковый контент из разных источников имеет одинаковый хеш. Поддержка отката через холодное хранилище (`live_vector_lake.py:169-209`). Разделение горячего/холодного снижает затраты на хранение Qdrant.

**Отрицательные:** Вычисление SHA-256 добавляет ~5ms на чанк во время ETL. Холодное хранилище (файлы Parquet) растёт неограниченно; очистка ручная (`cleanup_old_versions` в `hash_versioning.py:236-252`). Коллизии хешей теоретически возможны, но практически незначительны с SHA-256. Повреждение WAL требует ручного вмешательства (`--reset-wal` флаг в `run_etl.py:318`).

**Смягчения:** `force_reindex` (`run_etl.py:317`) обходит WAL для аварийного восстановления. Версионно-зависимая дедупликация во время поиска (`proxy/app/context_builder.py`) разрешает конфликты одинаковых документов, предпочитая более новые версии или явные запросы пользователя.
