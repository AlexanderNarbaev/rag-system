# ADR-004: Паттерн OpenAI-совместимого прокси

**Статус:** Принято
**Дата:** 2026-06-22

## Контекст

RAG-система должна интегрироваться с существующими инструментами в корпоративной среде — OpenWebUI, OpenCode и
пользовательскими внутренними клиентами — все из которых используют API чат-завершений OpenAI. Прокси также поддерживает
мульти-провайдерные бэкенды (OpenAI, Anthropic, Ollama и универсальные OpenAI-совместимые эндпоинты) через подключаемый
слой адаптеров (`provider_adapter.py`), обеспечивая гибкость LLM-бэкенда без изменения логики оркестрации. Создание
пользовательского API потребовало бы адаптации каждого клиента, увеличивая нагрузку на сопровождение и фрагментацию.
Система должна поддерживать как потоковый (генерация токенов в реальном времени), так и непотоковый режимы, при этом
внутренне выполняя RAG (поиск, реранкинг, сборку контекста) перед вызовом LLM.

Рассмотренные альтернативы: **нативная интеграция LangChain** (привязывает клиентов к конкретному фреймворку), *
*пользовательский REST API** (требует изменений на стороне клиента), **сервис gRPC** (не поддерживается существующими
инструментами).

## Решение

**Реализовать OpenAI-совместимый прокси с использованием FastAPI, предоставляя `/v1/chat/completions`, `/v1/models`
и `/v1/health`.**

Прокси определён в `proxy/app/main.py` с Pydantic-моделями, соответствующими схеме OpenAI:

- `ChatCompletionRequest` (`main.py:93-103`) отражает поля OpenAI (model, messages, temperature, max_tokens, stream) и
  добавляет RAG-специфичные расширения: `rag_version` для версионно-привязанного поиска, `rag_force_refresh` для обхода
  кэша.
- `ChatCompletionResponse` (`main.py:111-117`) включает статистику использования, finish_reason и идентификаторы
  моделей.

Потоковая передача использует Server-Sent Events (`main.py:317-337`) — `text/event-stream` с чанками `data: {json}\n\n`,
завершаемыми `data: [DONE]\n\n`, что соответствует потоковому протоколу OpenAI.

Внутренний RAG-пайплайн (`main.py:137-208`) прозрачен для клиента: поиск, реранкинг, дедупликация и сборка контекста
происходят внутри прокси до вызова LLM. Непотоковые ответы кэшируются в Redis на 1 час (`main.py:207`, `ttl=3600`).

Проверка здоровья (`/v1/health` в `main.py:211-238`) сообщает статус Qdrant и эндпоинта LLM, возвращая 503 при
деградации — совместимо с Docker health checks и балансировщиками нагрузки.

## Последствия

**Положительные:** Полная замена любого OpenAI-клиента — существующие инструменты указывают на URL прокси и продолжают
работать без изменений. Единый прокси обрабатывает и логику RAG, и маршрутизацию LLM, сокращая сетевые переходы.
RAG-расширения (`rag_version`, `rag_force_refresh`) игнорируются стандартными клиентами OpenAI (дополнительные поля
молча принимаются).

**Отрицательные:** Не-RAG запросы всё равно проходят полный пайплайн (поиск, сборка контекста), если не выделены особо.
Прокси добавляет 200-500ms задержки для поиска+реранкинга перед началом генерации LLM. Режим одного воркера (
`main.py:377`, `workers=1`) ограничивает параллелизм для защиты общего состояния (эмбеддер, кэш).

**Смягчения:** Слой кэширования снижает задержку повторных запросов до <10ms. Классификация интентов на основе SLM (
`slm_router.py:116-122`) может обнаруживать приветствия/запросы без контекста и пропускать поиск.
Production-масштабирование может использовать Redis-backed кэш для включения нескольких воркеров.
