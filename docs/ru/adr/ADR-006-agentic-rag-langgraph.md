# ADR-006: Агентная оркестрация RAG с LangGraph

**Статус:** Принято
**Дата:** 2026-06-22

## Контекст

Простой RAG (один поиск, затем генерация) не справляется со сложными запросами, требующими многошагового рассуждения, недостаточных результатов поиска или сравнения документов. Пользователи часто формулируют вопросы неоднозначно, и начальный поиск на основе эмбеддингов может вернуть чанки низкого качества. Система должна самокорректироваться, обнаруживая низкое качество поиска и итеративно уточняя как запрос, так и найденный контекст.

Рассмотренные альтернативы: **цепочки LangChain** (линейные, без условных циклов), **пользовательская машина состояний** (переизобретение потока управления, сложнее отлаживать), **рассуждение только LLM** (без цикла поиска, риск галлюцинаций), **CrewAI/Multi-agent** (излишне сложно для текущих паттернов запросов).

## Решение

**Реализовать агентную оркестрацию на основе LangGraph как опциональный модуль с fallback к стандартному RAG.**

Оркестратор (`proxy/app/orchestrator.py`) определяет граф состояний (`RAGState` в `orchestrator.py:36-51`) с семью узлами:
1. **rewrite** (`orchestrator.py:53-80`): LLM переписывает запрос для лучшего поиска.
2. **retrieve** (`orchestrator.py:83-105`): гибридный поиск в Qdrant.
3. **check_sufficiency** (`orchestrator.py:127-146`): оценивает качество поиска, используя порог среднего score (0.6), и итеративно возвращается к rewrite при недостаточности — ограничено `MAX_RETRIEVAL_LOOPS` (по умолчанию 3, `config.py:53`).
4. **graph_expand** (`orchestrator.py:108-124`): опционально обогащает контекст через обход графа сущностей Neo4j.
5. **rerank** (`orchestrator.py:149-167`): cross-encoder переоценка найденных чанков.
6. **build_context** (`orchestrator.py:170-182`): собирает финальный контекст с данными графа.
7. **generate** (`orchestrator.py:185-209`): генерирует финальный ответ с помощью LLM.

Граф компилируется с чекпоинтингом `MemorySaver` (`orchestrator.py:258`), обеспечивая сохранение состояния между итерациями цикла. LangGraph включается через переменную окружения `USE_LANGGRAPH` (`config.py:52`). При отключении прокси использует стандартный линейный пайплайн в `main.py:137-208`.

## Последствия

**Положительные:** Самокорректирующиеся циклы поиска улучшают качество ответов для неоднозначных запросов. Оценка достаточности предотвращает генерацию из некачественного контекста. Расширение графа через Neo4j (`orchestrator.py:118`) добавляет контекст связей сущностей, не охватываемый векторным поиском. Модульный дизайн — каждый узел можно отлаживать независимо.

**Отрицательные:** До 3× увеличения задержки (циклы rewrite + re-retrieve + re-rank). LangGraph добавляет сложность зависимостей и ~200 MB к Docker-образу. Расширение графа требует работающего Neo4j вместе с Qdrant. Fallback к простому RAG означает снижение качества без LangGraph.

**Смягчения:** `MAX_RETRIEVAL_LOOPS=3` ограничивает максимальную задержку. `check_sufficiency` использует быстрые эвристики (порог score), а не оценку LLM, сохраняя низкие накладные расходы цикла. LangGraph опционален — система работает в режиме простого RAG по умолчанию (`USE_LANGGRAPH=false` в `config.py:52`).
