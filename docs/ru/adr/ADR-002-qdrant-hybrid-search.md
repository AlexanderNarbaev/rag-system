# ADR-002: Qdrant для векторного хранения с гибридным поиском

**Статус:** Принято
**Дата:** 2026-06-22

## Контекст

RAG-системе требуется векторная база данных, поддерживающая гибридный поиск (dense semantic + sparse lexical), обрабатывающая инкрементальные обновления документов с версионированием и работающая в автономной Docker-среде. Технические документы из Confluence, Jira и GitLab требуют как точного совпадения по ключевым словам (номера тикетов, идентификаторы кода), так и семантического понимания.

Рассмотренные альтернативы: **Milvus** (более тяжёлое развёртывание, незрелая поддержка sparse на момент решения), **Weaviate** (гибридный поиск, но API только GraphQL, дополнительная сложность), **Pinecone** (SaaS — несовместим с требованием air-gapped), **pgvector** (расширение PostgreSQL — нет нативной поддержки sparse, ограниченный гибридный RRF). **Elasticsearch** исключён из-за отсутствия нативных возможностей плотных векторов без отдельных плагинов.

## Решение

**Использовать Qdrant как основное векторное хранилище с гибридным поиском через Reciprocal Rank Fusion (RRF).** Коллекция настроена с двумя типами векторов: `dense` (1024-dim, косинусное расстояние) и `sparse` (индекс на диске) — конфигурация в `etl/indexer/qdrant_hybrid.py:113-131`.

RRF-фьюжн объединяет списки результатов dense и sparse в `proxy/app/retrieval.py:113-128`, используя `k=60` для баланса смещения рангов. Разреженный индекс хранится на диске (`SparseIndexParams(on_disk=True)` в `qdrant_hybrid.py:118-121`) для снижения использования RAM на прокси-машине.

Создание коллекции поддерживает версионно-зависимые фильтрованные запросы через `FieldCondition` Qdrant по полю `version` (`retrieval.py:146-148`), что позволяет извлекать конкретные версии документов по запросу.

Квантизация (binary/scalar) и настройка HNSW (`ef_construct`, `m`) отложены до значений по умолчанию; стандартные настройки Qdrant обеспечивают адекватную производительность для <1M чанков.

## Последствия

**Положительные:** Единое развёртывание для плотного и разреженного поиска, не требуется отдельный индекс BM25. Индекс sparse на диске удерживает RAM под 8 GB для прокси-машины. REST API (`qdrant-client` HTTP на порту 6333) упрощает интеграцию с FastAPI-прокси. Инкрементальный upsert по хешу чанка (`qdrant_hybrid.py:186`, `id=point_id`) позволяет обновлять документы без полной переиндексации.

**Отрицательные:** Qdrant не имеет встроенного cross-encoder реранкинга — наш пайплайн добавляет его отдельно через `proxy/app/rerank.py`. Нет нативного обхода графа (решается интеграцией Neo4j). Дисковый sparse-индекс медленнее, чем в памяти, для очень больших коллекций (>10M точек).

**Смягчения:** Cross-encoder реранкинг после поиска компенсирует ограничения RRF-фьюжна. Слой кэширования (`proxy/app/cache.py`) снижает повторяющиеся одинаковые запросы. Мониторинг через эндпоинт `/v1/health` (`main.py:211-238`) проверяет подключение к Qdrant при каждой проверке здоровья.
