# ADR-003: Двухмодельная архитектура (SLM + LLM)

**Статус:** Принято
**Дата:** 2026-06-22

## Контекст

RAG-система обслуживает две различные категории рабочих нагрузок инференса: лёгкие задачи предобработки (классификация
интентов, декомпозиция запросов, извлечение сущностей) и тяжёлые задачи генерации (контекстный синтез ответов из
найденных чанков). Выполнение всех задач на одной большой модели тратит вычислительные ресурсы и добавляет задержку для
операций предобработки. Система должна работать в автономной среде с ограниченной ёмкостью GPU (одна машина с одним или
двумя GPU).

Рассмотренные альтернативы: **одна большая модель** (запуск всего на одной модели — высокая задержка для предобработки,
блокировка GPU для всех запросов), **маршрутизация только через API** (отклонено из-за air-gap), **предобработка на
основе правил** (недостаточная точность для многошаговой декомпозиции запросов).

## Решение

**Использовать двухмодельную архитектуру: малую языковую модель (SLM) для лёгких задач маршрутизации и основную большую
языковую модель (LLM) для генерации (например, Gemma-2B-it и Gemma-4-26B-it).**

SLM (`SLM_MODEL_NAME` в `proxy/app/config.py:39`) обрабатывает:

- Классификацию интентов (`proxy/app/slm_router.py:67-87`): классифицирует запросы на фактические, процедурные,
  сравнительные, summarization или приветствия.
- Декомпозицию запросов (`slm_router.py:90-113`): разбивает сложные запросы до 3 подзапросов.
- Извлечение сущностей (`slm_router.py:143-164`): извлекает технологии, названия проектов, номера тикетов.
- Лёгкое переписывание запросов (`slm_router.py:125-140`).

LLM (`LLM_MODEL_NAME` в `config.py:31`) обрабатывает генерацию через `proxy/app/llm_router.py` через OpenAI-совместимый
API (например, сервер `llama.cpp` или `vLLM`). Обе модели обслуживаются с одного инференс-сервера (`LLM_ENDPOINT`),
различаясь по имени модели.

Fallback: если `SLM_ENDPOINT` пуст (`config.py:38`), система использует эвристики — сопоставление ключевых слов для
интентов, regex для извлечения сущностей (`slm_router.py:41` возвращает пустую строку с предупреждением).

## Последствия

**Положительные:** SLM (~2 GB VRAM) работает вместе с LLM (чей VRAM зависит от размера модели и квантизации) на одном
GPU с батчингом. Задержка предобработки ~100ms (SLM) против ~800ms (LLM). Независимое масштабирование — SLM остаётся на
CPU, если GPU полностью занят генерацией LLM.

**Отрицательные:** Две модели для поддержки и обновления. Промпты должны версионироваться для обеих моделей отдельно.
Качество SLM снижается для пограничных запросов; эвристический fallback груб.

**Смягчения:** Задачи SLM ограничены структурированными выводами (JSON/enums) для валидации (`slm_router.py:102-111`).
Оркестратор (`proxy/app/orchestrator.py`) может использовать SLM или LLM для переписывания запросов, настраивается через
`MAX_RETRIEVAL_LOOPS` (`config.py:53`).
