# ADR-011: Инкрементальная/прогрессивная архитектура

**Статус:** Принято
**Дата:** 2026-07-10

## Контекст

RAG-система обслуживает корпоративных пользователей с различными средами развёртывания — от изолированных
онпрем-серверов до облачных кластеров. Не каждому развёртыванию нужен полный набор функций: небольшая команде,
тестирующей RAG-концепции, не требуется расширение графов Neo4j, кэширование Redis или агентная оркестрация LangGraph. В
то же время крупные продакшн-развёртывания требуют совместной работы всех этих компонентов.

Исследование конкурентных RAG-решений (2026-06-26) показало, что большинство open-source RAG-фреймворков попадают в две
крайности: либо слишком простые (базовый векторный поиск + LLM, без гибридного поиска, без реранжирования), либо слишком
сложные (обязательный Kubernetes, обязательная графовая БД, крутая кривая обучения). Ни одна крайность не служит
паттерну постепенного принятия, который нужен нашим пользователям.

Ключевые результаты исследования:

- **RAG best practices**: Гибридный поиск (dense + sparse) с RRF-слиянием стабильно превосходит чистый векторный поиск
  на 15-25% на бенчмарках поиска технических документов. Cross-encoder реранжирование добавляет ещё 10-15% точности
  ценой задержки.
- **Самоулучшающийся RAG**: Циклы обратной связи (HITL-коррекция → обогащение → переиндексация) создают порочный круг, в
  котором качество системы улучшается со временем без ручного вмешательства. Однако это требует опциональной сложности —
  цикл обратной связи не должен блокировать базовое использование.
- **Конкурентный анализ**: Продакшн RAG-системы, которые добиваются успеха, разделяют паттерн: простое ядро с
  опциональными слоями сложности. Системы, которые требуют все компоненты с самого начала, имеют более высокие
  показатели отказа при оценке.

Существующая кодовая база уже неформально отражает этот паттерн — флаги `USE_LANGGRAPH`, `GRAPH_ENABLED`, `USE_REDIS` —
но нет архитектурного решения, документирующего, почему этот паттерн был выбран и как он должен направлять дальнейшую
разработку.

## Решение

**Принять явную инкрементальную/прогрессивную архитектуру, в которой система работает с минимальным ядром (FastAPI +
Qdrant + LLM), а каждый дополнительный компонент является опциональным и корректно деградирует при отсутствии.**

Архитектура имеет три уровня:

### Уровень 1 — Минимальное ядро (всегда обязательно)

- FastAPI прокси с OpenAI-совместимым эндпоинтом `/v1/chat/completions`
- Qdrant для векторного поиска (dense эмбеддинги)
- Любой OpenAI-совместимый LLM-бэкенд (vLLM, llama.cpp или удалённый API)
- Базовая конфигурация через переменные окружения

Этот уровень обеспечивает работающую RAG-систему без дополнительной инфраструктуры.

### Уровень 2 — Улучшенный поиск (opt-in, рекомендуется для продакшна)

- Гибридный поиск: dense + sparse вектора с RRF-слиянием (sparse кодирование `BAAI/bge-m3`)
- Cross-encoder реранжирование (`MiniLM-L-6-v2`)
- Кэширование Redis для эмбеддингов и ответов
- Версионная индексация документов с WAL-чекпоинтингом
- Оптимизация токенов с BPE-осведомлённым распределением бюджета

Включается через: `USE_REDIS=true`, явная конфигурация sparse-векторов Qdrant.

### Уровень 3 — Продвинутая оркестрация (opt-in, для зрелых развёртываний)

- Агентный пайплайн LangGraph с вызовом инструментов
- Расширение графа Neo4j для многосвязных отношений сущностей
- SLM-маршрутизатор для классификации намерений и декомпозиции запросов
- Цикл HITL-обратной связи с самообогащением
- Пайплайн эволюции моделей (LoRA файн-тюнинг, канареечное развёртывание)

Включается через: `USE_LANGGRAPH=true`, `GRAPH_ENABLED=true`, отдельная точка доступа SLM.

Каждый уровень развёртывается независимо. Компоненты выходят из строя независимо — падение Neo4j не мешает chat
completion; недоступность Redis переключает на кэш в памяти; ошибки LangGraph откатывают к простому RAG-пайплайну.

Свидетельства реализации в кодовой базе:

- `proxy/app/main.py:78-80` — флаги `USE_LANGGRAPH`, `USE_REDIS` управляют активацией функций
- `proxy/app/core/retrieval.py` — расширение графа обёрнуто в guards `if GRAPH_ENABLED`
- `proxy/app/shared/cache.py` — CacheManager корректно деградирует при отсутствии URL Redis
- `proxy/app/core/orchestrator.py` — пайплайн LangGraph с fallback к простому поиску
- `proxy/app/shared/config.py` — все продвинутые функции по умолчанию `False`

## Последствия

**Положительные:**

- **Низкий порог входа**: Новые пользователи могут запустить `docker-compose up` с минимальным `.env` и получить
  работающую RAG-систему за считанные минуты.
- **Инкрементальное принятие**: Команды могут включать функции по мере роста — начать с Уровня 1, добавить Redis и
  реранжирование, когда важна задержка, добавить расширение графа, когда важны связи сущностей.
- **Отказоустойчивость по дизайну**: Отсутствие каждого компонента — это тестируемый путь кода, а не непроверенный
  граничный случай. Набор тестов хаоса/отказоустойчивости (`tests/resilience/`) проверяет корректную деградацию для
  каждого опционального компонента.
- **Совместимость с air-gapped**: Уровень 1 требует только одной загрузки модели (эмбеддер). Более высокие уровни
  добавляют модели инкрементально.
- **Чёткий путь обновления**: Каждый уровень имеет явные флаги конфигурации, что упрощает документирование и
  автоматизацию профилей развёртывания.

**Отрицательные:**

- **Сложность матрицы тестирования**: С N опциональными функциями количество комбинаций конфигурации составляет 2^N.
  Смягчается тестированием Уровня 1 (всё выключено) и Уровня 3 (всё включено) как основных целей, с точечными тестами
  для отдельных переключателей функций.
- **Накладные расходы ветвления кода**: Каждый опциональный компонент требует условных guards в горячем пути. Смягчается
  изоляцией guards на границах модулей (например, `retrieval.py` обрабатывает свой guard расширения графа внутри себя).
- **Нагрузка на документацию**: Каждый уровень нуждается в чёткой документации о том, что включено, что требуется и что
  деградирует. Смягчается профилями конфигурации по уровням в `docs/en/guides/deployment-guide.md`.

**Смягчения:**

- Конфигурация по умолчанию поставляется как Уровень 1 (все продвинутые функции выключены) — новые развёртывания
  получают простейшую работающую систему.
- `tests/resilience/test_chaos.py` явно тестирует отсутствие каждого компонента (Qdrant недоступен, Redis недоступен,
  Neo4j недоступен, таймаут LLM, комбинированные сбои).
- Модуль `proxy/app/shared/config.py` централизует все флаги функций с разумными значениями по умолчанию и
  документацией.
