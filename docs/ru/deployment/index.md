# Развёртывание

RAG-система разворачивается как два отдельных компонента, взаимодействующих по сети:

1. **RAG Proxy** — стек Docker Compose с прокси, Qdrant, Neo4j, Redis, LLM-бэкендом и HITL-панелью
2. **ETL Pipeline** — Python-приложение, запускаемое по расписанию для извлечения и индексации данных

## Руководства

| Руководство                                | Описание                                                                          |
|--------------------------------------------|-----------------------------------------------------------------------------------|
| [Развёртывание прокси](../deploy_proxy.md) | Настройка Docker Compose, конфигурация, масштабирование, автономное развёртывание |
| [Развёртывание ETL](../deploy_etl.md)      | Конфигурация пайплайна, планирование, настройка систем-источников                 |

## Архитектура

```
┌──────────────────────┐         ┌──────────────────────┐
│    ETL-машина         │         │   Прокси-машина       │
│                       │         │                       │
│  extractors/ ──┐      │         │  ┌─────────────────┐  │
│  chunker/     │      │   API   │  │   rag-proxy     │  │
│  graph_builder/      │◄────────┼──│   :8080         │  │
│  indexer/ ─────┘      │         │  └───────┬─────────┘  │
│                       │         │          │            │
│  → запись в Qdrant,   │         │  ┌───────┴─────────┐  │
│    Neo4j через API     │         │  │ qdrant  :6333   │  │
│                       │         │  │ redis   :6379   │  │
│                       │         │  │ neo4j   :7687   │  │
│                       │         │  │ llm     :8000   │  │
│                       │         │  └─────────────────┘  │
└──────────────────────┘         └──────────────────────┘
```

ETL-машина записывает данные в те же экземпляры Qdrant и Neo4j, из которых читает прокси. В типичной конфигурации оба
компонента работают на разных машинах, но используют общие эндпоинты Qdrant и Neo4j через внутреннюю сеть.
