## Обзор

RAG System следует модели безопасности «глубокой эшелонированной защиты» с множественными уровнями:

| Уровень                     | Компонент                                | Файл                               |
|-----------------------------|------------------------------------------|------------------------------------|
| **Аутентификация**          | JWT-токены, Keycloak OIDC, LDAP/AD       | `proxy/app/auth/jwt.py`            |
| **Авторизация**             | Контроль доступа на основе ролей (RBAC)  | `proxy/app/auth/rbac.py`           |
| **Хранилище пользователей** | SQLite с bcrypt-паролями                 | `proxy/app/auth/user_db.py`        |
| **LDAP-интеграция**         | Active Directory / LDAP bind             | `proxy/app/auth/ldap.py`           |
| **Санитизация ввода**       | SQL-инъекции, XSS, ограничение длины     | `proxy/app/shared/sanitizer.py`    |
| **Утилиты безопасности**    | Валидация, секреты, заголовки            | `proxy/app/shared/security.py`     |
| **Ограничение частоты**     | Token bucket по IP/ключу API             | `proxy/app/shared/rate_limiter.py` |
| **Аудит-логирование**       | Событийный лог JSONL                     | `proxy/app/shared/audit.py`        |
| **Промежуточное ПО**        | Request ID, CORS, заголовки безопасности | `proxy/app/shared/middleware.py`   |

### Принципы проектирования

- **Изолированная среда прежде всего**: никаких внешних API-вызовов во время работы. Все модели предварительно
  загружены.
- **Постепенная деградация**: каждый компонент может отказать независимо без падения прокси.
- **Глубокая эшелонированная защита**: множество независимых уровней безопасности — даже при отказе одного другие
  защищают.
- **Принцип минимальных привилегий**: RBAC обеспечивает минимально необходимые разрешения для каждой роли.
- **Нулевое доверие к вводу**: все пользовательские данные санитизируются перед обработкой.

## Аутентификация

### JWT-аутентификация

Система поддерживает два алгоритма JWT:

| Алгоритм                 | Применение                               | Источник ключа                     |
|--------------------------|------------------------------------------|------------------------------------|
| **HS256** (по умолчанию) | Локальная разработка, сервисные аккаунты | Переменная `JWT_SECRET`            |
| **RS256**                | Keycloak OIDC, продакшн SSO              | `JWT_PUBLIC_KEY` или JWKS-эндпоинт |

#### Конфигурация

```bash
AUTH_ENABLED=true                    # Включить аутентификацию (по умолчанию: false)
JWT_SECRET=your-secret-key           # Ключ подписи HS256
JWT_ALGORITHM=HS256                  # или RS256
JWT_PUBLIC_KEY=""                    # PEM-ключ для RS256
TOKEN_EXPIRE_HOURS=24                # Время жизни токена
ACCESS_TOKEN_MINUTES=60              # Время жизни access-токена
REFRESH_TOKEN_DAYS=7                 # Время жизни refresh-токена
```

#### Структура токена

```json
{
  "sub": "user-uuid",
  "preferred_username": "alice",
  "roles": ["user"],
  "groups": ["engineering"],
  "access_level": "internal",
  "namespace": "engineering",
  "iat": 1699900000,
  "exp": 1700000000,
  "jti": "unique-token-id"
}
```

#### Эндпоинты аутентификации

| Эндпоинт            | Метод | Описание                                                        |
|---------------------|-------|-----------------------------------------------------------------|
| `/v1/auth/login`    | POST  | Вход по логину/паролю, возвращает пару access + refresh токенов |
| `/v1/auth/register` | POST  | Самостоятельная регистрация (bcrypt-пароли в SQLite)            |
| `/v1/auth/refresh`  | POST  | Обмен refresh-токена на новую пару access + refresh             |
| `/v1/auth/logout`   | POST  | Отзыв refresh-токенов, добавление access-токена в чёрный список |
| `/v1/auth/me`       | GET   | Текущий контекст пользователя                                   |

#### Режимы аутентификации

1. **Отключена** (`AUTH_ENABLED=false`, по умолчанию): все запросы используют анонимный контекст с полным доступом.
2. **Включена** (`AUTH_ENABLED=true`): требуется валидный Bearer-токен для всех `/v1/*` эндпоинтов.
3. **Keycloak OIDC** (установлен `KEYCLOAK_URL`): автоматическое обнаружение конфигурации OIDC, валидация RS256-токенов
   через JWKS.

#### Публичные эндпоинты (без аутентификации даже при включённой)

- `/v1/auth/login`, `/v1/auth/register`, `/v1/auth/refresh`
- `/v1/health`, `/v1/health/live`, `/v1/health/ready`
- `/v1/models`, `/v1/widget`, `/v1/widget.js`
- `/metrics`

### Интеграция с Keycloak OIDC

При настройке `KEYCLOAK_URL`:

```bash
KEYCLOAK_URL=https://keycloak.example.com
KEYCLOAK_REALM=master
KEYCLOAK_CLIENT_ID=rag-proxy
```

Система автоматически:

1. Получает JWKS с `{KEYCLOAK_URL}/realms/{REALM}/protocol/openid-connect/certs`
2. Кеширует ключи на 1 час
3. Валидирует RS256-токены с помощью полученных открытых ключей
4. Возвращается к `JWT_PUBLIC_KEY` если JWKS недоступен (изолированная среда)

### LDAP / Active Directory

При `AD_ENABLED=true` эндпоинт входа сначала пытается LDAP bind, а затем возвращается к локальной аутентификации SQLite.

```bash
AD_ENABLED=true
AD_URL=ldap://ldap.example.com:389
AD_BASE_DN=dc=example,dc=com
AD_USER_DN_TEMPLATE=cn={username},{base_dn}
```

**Поведение:**

- При успешном LDAP bind пользователь автоматически создаётся в локальной базе SQLite с ролью `user` по умолчанию.
- Если LDAP-сервер недоступен, происходит переход к локальной аутентификации с предупреждением в логе.
- LDAP-пользователи получают случайный локальный пароль (не используется для аутентификации).

## Авторизация (RBAC)

### Роли

Роли иерархические — старшие роли наследуют все разрешения младших:

| Роль        | Ранг | Разрешения                                                      |
|-------------|------|-----------------------------------------------------------------|
| `admin`     | 4    | Все эндпоинты, управление пользователями, конфигурация, прогрев |
| `expert`    | 3    | Чат, отправка/рецензирование обратной связи, запуск обогащения  |
| `user`      | 2    | Чат, стриминг, доступ к виджету                                 |
| `read_only` | 1    | Список моделей, проверки здоровья, эндпоинты аутентификации     |

### Конфигурация

```bash
RBAC_ENABLED=true    # Включить RBAC (по умолчанию: false)
```

При отключённом RBAC все пользователи имеют полный доступ независимо от роли.

### Карта разрешений

```
admin:config      → ADMIN
admin:users       → ADMIN
admin:stats       → ADMIN
feedback          → EXPERT
feedback:submit   → EXPERT
feedback:review   → EXPERT
enrichment:trigger → EXPERT
chat              → USER
chat:stream       → USER
widget:access     → USER
models:list       → READ_ONLY
health:check      → READ_ONLY
auth:login        → READ_ONLY
auth:register     → READ_ONLY
```

### Использование в коде

```python
from proxy.app.auth.rbac import Role, require_role

@app.post("/v1/feedback")
async def submit_feedback(user: UserContext = Depends(require_role(Role.EXPERT))):
    # Только эксперты и администраторы могут достичь этой точки
    ...
```

## Санитизация входных данных

### Модуль санитизации (`proxy/app/shared/sanitizer.py`)

Предоставляет три функции санитизации:

#### `sanitize_query(text)`

Удаляет из пользовательских поисковых запросов:

- Ключевые слова SQL/DQL-инъекций (`SELECT`, `INSERT`, `UPDATE`, `DELETE`, `DROP` и др.)
- SQL-комментарии (`--`)
- Точки с запятой
- HTML-теги
- Протоколы JavaScript/VBScript
- Обработчики событий (`onclick`, `onload` и др.)
- CSS-выражения
- Управляющие символы
- Максимальная длина: 8000 символов

#### `sanitize_feedback(text)`

Удаляет из экспертной обратной связи:

- Теги script, iframe
- HTML-теги
- Протоколы JavaScript/VBScript/data
- Обработчики событий
- CSS-выражения
- XSS с кодированием сущностей (`&#x...;`)
- Управляющие символы
- Максимальная длина: 32000 символов

#### `validate_length(text, max_len)`

Обрезает текст до указанной максимальной длины.

### Утилиты безопасности (`proxy/app/shared/security.py`)

#### InputValidator

| Метод                            | Описание                                                               |
|----------------------------------|------------------------------------------------------------------------|
| `validate_query(query)`          | Санитизация пользовательского запроса (макс. 8000 символов)            |
| `validate_non_empty(s, max_len)` | Валидация непустой строки в пределах длины                             |
| `sanitize_for_log(text)`         | Маскирование email, IP, API-ключей для безопасного логирования         |
| `validate_model_name(name)`      | Проверка имени модели на допустимые символы                            |
| `validate_path_traversal(path)`  | Проверка на обход пути (`..`, `~`, `\x00`)                             |
| `sanitize_headers(headers)`      | Редактирование чувствительных заголовков (Authorization, Cookie и др.) |
| `escape_shell_arg(arg)`          | Удаление небезопасных для оболочки символов                            |

#### SecretsManager

| Метод                           | Описание                                                      |
|---------------------------------|---------------------------------------------------------------|
| `generate_api_key(prefix)`      | Генерация криптографически стойкого API-ключа                 |
| `hash_secret(secret)`           | SHA-256 хеш для хранения                                      |
| `verify_secret(secret, hashed)` | Проверка по хешу                                              |
| `mask_in_response(data)`        | Глубокое маскирование чувствительных полей в ответах          |
| `generate_token(length)`        | Случайный URL-безопасный токен                                |
| `constant_time_compare(a, b)`   | Сравнение строк с постоянным временем (защита от timing-атак) |

### Санитизация на уровне промежуточного ПО

`InputSanitizationMiddleware` автоматически санитизирует параметры запроса при каждом обращении.

## Ограничение частоты запросов

### Алгоритм Token Bucket

Ограничитель частоты использует алгоритм token bucket с отслеживанием по IP и ключу API.

#### Конфигурация

```bash
RATE_LIMIT_ENABLED=true       # Включить ограничение частоты (по умолчанию: false)
RATE_LIMIT_PER_MINUTE=60      # Запросов в минуту на ключ
RATE_LIMIT_BURST=10           # Максимальный размер пакета
```

#### Поведение

1. Каждый IP-адрес клиента (или ключ API из Bearer-токена) получает независимый bucket.
2. Токены пополняются со скоростью `RATE_LIMIT_PER_MINUTE / 60` в секунду.
3. Ёмкость bucket — `RATE_LIMIT_BURST` токенов.
4. При исчерпании возвращается HTTP 429 с заголовком `Retry-After`.
5. Истёкшие bucket (не использовавшиеся более 5 минут) автоматически очищаются.

#### Извлечение ключа

- Если присутствует `Authorization: Bearer <token>` → ключ `apikey:<token>`
- Иначе → ключ `ip:<client-ip>` (учитывает `X-Forwarded-For`)

#### Ответ

```json
// HTTP 429
{
  "error": "Rate limit exceeded"
}
// Заголовки: Retry-After: <секунды>
```

## Аудит-логирование

### Конфигурация

```bash
AUDIT_ENABLED=true    # Включить аудит-логирование (по умолчанию: true)
LOG_DIR=./logs        # Директория логов аудита
```

### Типы событий

| Тип события     | Описание                                 | Логируемые поля                                                    |
|-----------------|------------------------------------------|--------------------------------------------------------------------|
| `query`         | Выполнение RAG-запроса                   | user_id, query_preview, response_preview, chunks, duration, tokens |
| `login`         | Попытка аутентификации                   | user_id, action, success                                           |
| `auth`          | События аутентификации (logout, refresh) | user_id, action, success                                           |
| `access_denied` | Отказ в доступе                          | user_id, resource, reason                                          |
| `config_change` | Изменение конфигурации                   | key, old_value (маскировано), new_value (маскировано)              |
| `error`         | Ошибки приложения                        | error_type, message, stack_trace                                   |
| `trace`         | Детальная трассировка запроса            | query, chunks, оценки rerank, разбивка токенов, уверенность        |

### Формат лога

События аудита записываются в `/var/log/rag-system/audit.jsonl` как JSON Lines:

```json
{
  "event_id": "evt_1699900000_abc123",
  "timestamp": "2024-01-15T10:30:00+00:00",
  "event_type": "query",
  "user_id": "alice",
  "client_ip": "10.0.0.1",
  "endpoint": "/v1/chat/completions",
  "request_hash": "a1b2c3d4e5f6g7h8",
  "details": {
    "query_preview": "How to configure...",
    "chunks_retrieved": 5,
    "metadata": {}
  },
  "duration_ms": 1234.56,
  "tokens_used": 1500,
  "result_status": "success"
}
```

### Обработка чувствительных данных

- Содержимое запросов обрезается до 200 символов в логах аудита
- Значения конфигурации маскируются (первые 4 + последние 4 символа)
- Хеши запросов используют SHA-256 (первые 16 hex-символов)

## Управление секретами

### Переменные окружения

Все секреты настраиваются через переменные окружения или файл `.env`:

| Переменная         | Назначение                   |
|--------------------|------------------------------|
| `JWT_SECRET`       | Ключ подписи JWT             |
| `JWT_PUBLIC_KEY`   | Открытый ключ RS256 (PEM)    |
| `LLM_API_KEY`      | API-ключ бэкенда LLM         |
| `EMBEDDER_API_KEY` | API-ключ сервиса эмбеддингов |
| `RERANKER_API_KEY` | API-ключ сервиса rerank      |
| `NEO4J_PASSWORD`   | Пароль базы данных Neo4j     |
| `AD_BASE_DN`       | Базовый DN LDAP              |

### Рекомендации

1. **Никогда не коммитьте файлы `.env`** в систему контроля версий.
2. **Используйте разные секреты** для разных окружений (dev/staging/prod).
3. **Регулярно ротируйте секреты** — система поддерживает горячую перезагрузку конфигурации.
4. **Маскируйте секреты в логах** — модуль логирования автоматически маскирует API-ключи, пароли и токены.
5. **Используйте Kubernetes Secrets** или HashiCorp Vault в продакшне.

### Маскирование в логах

Модуль логирования (`proxy/app/shared/logging.py`) автоматически маскирует:

- Значения `api_key`, `API_KEY`
- Токены `Authorization: Bearer`
- Значения `password`
- Значения `secret`
- Значения `token`

## Сетевая безопасность

### Конфигурация CORS

```bash
CORS_ORIGINS=*                    # Разрешённые источники (через запятую или *)
```

Промежуточное ПО CORS暴露ирует эти заголовки клиентам:

- `X-Request-ID`
- `X-Correlation-ID`
- `Retry-After`

### Заголовки безопасности

`SecurityHeadersMiddleware` добавляет эти заголовки в каждый ответ:

| Заголовок                   | Значение                                   |
|-----------------------------|--------------------------------------------|
| `X-Content-Type-Options`    | `nosniff`                                  |
| `X-Frame-Options`           | `DENY`                                     |
| `X-XSS-Protection`          | `1; mode=block`                            |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains`      |
| `Content-Security-Policy`   | `default-src 'self'`                       |
| `Referrer-Policy`           | `strict-origin-when-cross-origin`          |
| `Permissions-Policy`        | `camera=(), microphone=(), geolocation=()` |
| `Cache-Control`             | `no-store, no-cache, must-revalidate`      |

### HTTPS

Сам прокси работает по HTTP. Завершайте TLS на обратном прокси (nginx, Traefik, Kubernetes Ingress).

## Развёртывание в изолированной среде

Система предназначена для работы без доступа в интернет:

1. **Все модели предварительно загружены** с помощью `scripts/download_models_offline.py`.
2. **Нет внешних API-вызовов** во время работы — получение JWKS Keycloak является необязательным и корректно
   обрабатывает ошибки.
3. **Локальные запасные варианты** — при недоступности удалённого embedder/reranker происходит возврат к локальным
   моделям.
4. **Нет телеметрии** — OpenTelemetry отключён по умолчанию и отправляет данные только в локальный коллектор.

## Чеклист безопасности

### Продакшн-развёртывание

- [ ] `AUTH_ENABLED=true` — аутентификация включена
- [ ] `RBAC_ENABLED=true` — контроль доступа на основе ролей активен
- [ ] `RATE_LIMIT_ENABLED=true` — ограничение частоты запросов активно
- [ ] `SANITIZE_INPUT=true` — санитизация ввода включена
- [ ] `AUDIT_ENABLED=true` — аудит-логирование активно
- [ ] `JWT_SECRET` установлен на стойкое случайное значение (32+ символов)
- [ ] `JWT_ALGORITHM=RS256` с Keycloak OIDC для продакшн SSO
- [ ] `CORS_ORIGINS` ограничен конкретными доменами (не `*`)
- [ ] TLS завершается на обратном прокси
- [ ] Файл `.env` не закоммичен в систему контроля версий
- [ ] Секреты ротируются регулярно
- [ ] Логи аудита мониторятся и хранятся
- [ ] `WORKERS=1` для защиты общего состояния (embedder, кеш)
- [ ] `LOG_FORMAT=json` для структурированной агрегации логов
- [ ] Эндпоинты здоровья доступны для Kubernetes-проб

### Разработка

- [ ] `AUTH_ENABLED=false` для локальной разработки
- [ ] Используйте `create_mock_token()` для тестирования
- [ ] Ограничение частоты отключено для нагрузочного тестирования
- [ ] Аудит-логирование включено даже в dev для отладки
