## Ежедневные операции

### Утренний чеклист (09:00)

- [ ] **Проверить работоспособность сервисов**
  ```bash
  curl -s http://localhost:8080/v1/health | jq .
  ```

- [ ] **Убедиться, что все контейнеры запущены**
  ```bash
  docker-compose -f proxy/docker-compose.yml ps
  ```

- [ ] **Проверить журналы ошибок на наличие аномалий**
  ```bash
  docker-compose -f proxy/docker-compose.yml logs --since 24h | grep -i error
  ```

- [ ] **Проконтролировать использование ресурсов**
  ```bash
  docker stats --no-stream
  ```

- [ ] **Подтвердить завершение резервного копирования**
  ```bash
  make verify-backups
  ```

### В течение рабочего дня

- **Мониторинг времени отклика** через метрики Prometheus по адресу `/metrics`
- **Проверка HITL-дашборда** на наличие экспертной обратной связи (http://localhost:8501)
- **Анализ метрик ограничения частоты запросов** на предмет необычных паттернов

### Конец дня (18:00)

- [ ] **Запустить ночное резервное копирование** (если не автоматизировано)
  ```bash
  make backup
  ```

- [ ] **Просмотреть сводку дневных метрик**

## Ежемесячный обзор

### Первый понедельник месяца

- [ ] **Обзор базовых показателей производительности**
    - Сравнение текущих метрик с предыдущим месяцем
    - Выявление трендов задержки, пропускной способности, частоты ошибок

- [ ] **Планирование ёмкости**
    - Анализ роста хранилищ (Qdrant, Neo4j, Redis)
    - Оценка сроков необходимости масштабирования
    - Проверка требований к хранилищу моделей

- [ ] **Аудит безопасности**
  ```bash
  # Проверка уязвимых зависимостей
  pip audit
  # Анализ журналов доступа
  grep "401\|403" logs/proxy-*.log
  ```

- [ ] **Обзор стратегии резервного копирования**
    - Проверка политики хранения копий (7 ежедневных, 4 еженедельных, 3 ежемесячных)
    - Тестирование полной процедуры восстановления
    - Обновление скриптов резервного копирования при необходимости

- [ ] **Обновление документации**
    - Добавление новых процедур в runbook
    - Просмотр и обновление контактов для эскалации

### Ежеквартально

- [ ] **Учения по аварийному восстановлению**
    - Имитация полного отказа системы
    - Практика восстановления из резервных копий
    - Документирование фактического времени восстановления vs. целевого

- [ ] **Оценка качества моделей**
    - Запуск конвейера оценки на отложенной тестовой выборке
    - Сравнение с метриками предыдущего квартала
    - Решение о необходимости дообучения

## Реагирование на инциденты

### Уровни серьёзности

| Уровень              | Описание                             | Время реагирования | Пример                                              |
|----------------------|--------------------------------------|--------------------|-----------------------------------------------------|
| **P1 — Критический** | Система недоступна, потеря данных    | 15 минут           | Все сервисы не отвечают, повреждение данных         |
| **P2 — Высокий**     | Неработоспособность ключевой функции | 1 час              | LLM не отвечает, поиск возвращает ошибки            |
| **P3 — Средний**     | Снижение производительности          | 4 часа             | Высокая задержка, периодические сбои                |
| **P4 — Низкий**      | Незначительная проблема              | 24 часа            | Визуальный дефект UI, некритичные ошибки в журналах |

### Процедура реагирования на инцидент

#### 1. Обнаружение и оповещение (0–5 минут)

```bash
# Быстрая проверка работоспособности
curl -s http://localhost:8080/v1/health

# Проверка всех сервисов
docker-compose -f proxy/docker-compose.yml ps

# Проверка использования ресурсов
docker stats --no-stream
```

#### 2. Оценка и коммуникация (5–15 минут)

- Определить уровень серьёзности
- Уведомить заинтересованные стороны через соответствующий канал
- Создать тикет инцидента

#### 3. Смягчение последствий (15–60 минут)

**Если LLM-бэкенд недоступен:**

```bash
# Проверить эндпоинт LLM
curl -s http://localhost:8000/v1/models

# Перезапустить сервис LLM
docker-compose -f proxy/docker-compose.yml restart llm
```

**Если Qdrant не отвечает:**

```bash
# Проверить статус Qdrant
curl -s http://localhost:6333/healthz

# Перезапустить Qdrant
docker-compose -f proxy/docker-compose.yml restart qdrant
```

**Если Redis недоступен:**

```bash
# Проверить Redis
docker exec redis redis-cli ping

# Перезапустить Redis
docker-compose -f proxy/docker-compose.yml restart redis
```

**Если Neo4j недоступен:**

```bash
# Проверить Neo4j
docker exec neo4j cypher-shell "RETURN 1;"

# Перезапустить Neo4j
docker-compose -f proxy/docker-compose.yml restart neo4j
```

#### 4. Устранение и документирование (1–4 часа)

- Выявить корневую причину
- Внедрить постоянное исправление
- Обновить мониторинг/оповещения при необходимости
- Задокументировать инцидент по шаблону постмортема

#### 5. Постинцидентный разбор (24–48 часов)

- Провести бесконфликтный постмортем
- Определить превентивные меры
- Обновить runbook на основе извлеченных уроков
- Поделиться результатами с командой

### Типичные сценарии инцидентов

#### Сценарий: Высокая задержка (время отклика >5 с)

```bash
# 1. Проверить использование ресурсов
docker stats --no-stream

# 2. Проверить нагрузку Qdrant
curl -s http://localhost:6333/metrics | grep qdrant_

# 3. Проверить LLM-бэкенд
time curl -s http://localhost:8000/v1/models

# 4. Перезапустить сервисы при необходимости
docker-compose -f proxy/docker-compose.yml restart
```

#### Сценарий: Утечка памяти (OOM-убийства)

```bash
# 1. Проверить лимиты памяти контейнера
docker inspect <container> | grep -A 5 "Memory"

# 2. Проверить наличие OOM-убийств
dmesg | grep -i "out of memory"

# 3. Перезапустить затронутый сервис
docker-compose -f proxy/docker-compose.yml restart <service>

# 4. Мониторить использование памяти
watch -n 1 'docker stats --no-stream'
```

#### Сценарий: Повреждение данных

```bash
# 1. Остановить операции записи
docker-compose -f proxy/docker-compose.yml stop proxy

# 2. Оценить масштаб повреждений
curl -s http://localhost:6333/collections | jq '.result.collections | length'

# 3. Восстановить из резервной копии
./scripts/ops/restore_all.sh

# 4. Проверить восстановление
make verify-backups

# 5. Возобновить работу
docker-compose -f proxy/docker-compose.yml start proxy
```

## Типичные проблемы

### Проблема: Прокси возвращает 502 Bad Gateway

**Симптомы**: Nginx/обратный прокси возвращает 502

**Диагностика:**

```bash
# Проверить, запущен ли прокси
docker-compose -f proxy/docker-compose.yml ps proxy

# Просмотреть журналы прокси
docker-compose -f proxy/docker-compose.yml logs proxy --tail=50

# Проверить привязку порта
netstat -tlnp | grep 8080
```

**Решение:**

```bash
# Перезапустить прокси
docker-compose -f proxy/docker-compose.yml restart proxy

# Если проблема сохраняется, проверить конфигурацию .env
cat proxy/.env | grep -E "^(HOST|PORT|WORKERS)"
```

### Проблема: Таймаут сервиса эмбеддингов

**Симптомы**: Медленные или неудачные запросы на создание эмбеддингов

**Диагностика:**

```bash
# Проверить работоспособность сервиса эмбеддингов
curl -s http://localhost:8080/v1/health | jq '.embedding_service'

# Проверить загрузку GPU (при наличии)
nvidia-smi
```

**Решение:**

```bash
# Перезапустить сервис эмбеддингов
docker-compose -f proxy/docker-compose.yml restart embedder

# При нехватке памяти GPU — уменьшить размер батча в конфигурации
# Изменить в proxy/.env: EMBEDDING_BATCH_SIZE=16
```

### Проблема: Коллекция Qdrant не найдена

**Симптомы**: Поиск возвращает пустые результаты

**Диагностика:**

```bash
# Список коллекций
curl -s http://localhost:6333/collections | jq .

# Информация о коллекции
curl -s http://localhost:6333/collections/{collection_name} | jq .
```

**Решение:**

```bash
# Переинициализировать коллекции
python scripts/init_collections.py

# Повторно запустить ETL, если данные отсутствуют
make etl
```

## Полезные команды

### Управление сервисами

```bash
# Запустить все сервисы
docker-compose -f proxy/docker-compose.yml up -d

# Остановить все сервисы
docker-compose -f proxy/docker-compose.yml down

# Перезапустить конкретный сервис
docker-compose -f proxy/docker-compose.yml restart <service>

# Просмотр журналов (с подпиской)
docker-compose -f proxy/docker-compose.yml logs -f <service>

# Просмотр журналов (последние 100 строк)
docker-compose -f proxy/docker-compose.yml logs --tail=100 <service>
```

### Проверки работоспособности

```bash
# Полная проверка работоспособности
curl -s http://localhost:8080/v1/health | jq .

# Проверка жизнеспособности (K8s)
curl -s http://localhost:8080/v1/health/live

# Проверка готовности (K8s)
curl -s http://localhost:8080/v1/health/ready
```

### Операции с базами данных

```bash
# Qdrant — Список коллекций
curl -s http://localhost:6333/collections | jq .

# Qdrant — Информация о коллекции
curl -s http://localhost:6333/collections/{name} | jq .

# Neo4j — Количество узлов
docker exec neo4j cypher-shell "MATCH (n) RETURN count(n);"

# Redis — Проверка ключей
docker exec redis redis-cli keys "*"
```

### Резервное копирование и восстановление

```bash
# Запустить все резервные копии
make backup

# Проверить резервные копии
make verify-backups

# Восстановить из резервной копии
make restore
```

### Развёртывание

```bash

make deploy

# Развернуть в production
make deploy-prod

# Получить последние образы
docker-compose -f proxy/docker-compose.yml pull
```

## Приложение

### Расположение журналов

- **Журналы прокси**: `docker-compose logs proxy`
- **Журналы ETL**: `logs/etl-*.log`
- **Системные журналы**: `/var/log/syslog`
- **Журналы приложения**: `logs/proxy-*.log`

### Файлы конфигурации

- **Основная конфигурация**: `proxy/.env`
- **Docker Compose**: `proxy/docker-compose.yml`
- **Конфигурация ETL**: `etl/config/etl_config.yaml`
- **Конфигурация Nginx**: `deploy/nginx/nginx.conf`

### URL-адреса мониторинга

- **Prometheus**: http://localhost:9090
- **Grafana**: http://localhost:3000
- **HITL-дашборд**: http://localhost:8501
- **Проверка работоспособности API**: http://localhost:8080/v1/health

### Ключевые метрики для наблюдения

| Метрика              | Порог предупреждения | Критический порог |
|----------------------|----------------------|-------------------|
| Время отклика (p95)  | >2 с                 | >5 с              |
| Частота ошибок       | >1%                  | >5%               |
| Загрузка CPU         | >70%                 | >90%              |
| Использование памяти | >80%                 | >95%              |
| Использование диска  | >75%                 | >90%              |
| Нагрузка Qdrant      | >70%                 | >90%              |
