# Руководство пользователя

Как использовать RAG Knowledge Assistant: аутентификация, роли, обратная связь и то, как ваш ввод улучшает систему.

## Обзор

RAG-система разработана для обучения на основе взаимодействий пользователей. Каждый запрос, каждая оценка ответа и каждое исправление эксперта возвращаются в систему для улучшения будущих результатов. В этом руководстве объясняется, как эффективно взаимодействовать с системой и как ваша обратная связь обеспечивает непрерывное улучшение.

## Роли пользователей

Система реализует ролевую модель доступа (RBAC) с четырьмя иерархическими ролями:

| Роль | Разрешения | Типичный пользователь |
|------|-----------|----------------------|
| **admin** | Все эндпоинты: чат, обратная связь, управление пользователями, конфигурация, метрики | Системные администраторы |
| **expert** | Чат + отправка отзывов + проверка + запуск обогащения | Доменные эксперты, кураторы знаний |
| **user** | Чат + доступ к виджету | Обычные пользователи |
| **read_only** | Список моделей + проверка здоровья + эндпоинты аутентификации | API-потребители, мониторинг |

Роли иерархичны — более высокая роль наследует все разрешения низших ролей. Когда RBAC отключён (`RBAC_ENABLED=false`), все аутентифицированные пользователи имеют полный доступ.

## Аутентификация

### Вход в систему

Отправьте `POST`-запрос на `/v1/auth/login`:

```bash
curl -X POST http://localhost:8080/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "alice", "password": "your-password"}'
```

Ответ:

```json
{
  "access_token": "eyJhbGciOiJIUzI1...",
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2g...",
  "token_type": "bearer",
  "expires_in": 3600
}
```

### Регистрация

Самостоятельная регистрация доступна по адресу `/v1/auth/register`:

```bash
curl -X POST http://localhost:8080/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username": "alice", "password": "secure-password", "email": "alice@example.com"}'
```

Новым пользователям по умолчанию назначается роль `user`.

### Использование токенов

Включите токен доступа в запросы через заголовок `Authorization`:

```bash
curl http://localhost:8080/v1/chat/completions \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1..." \
  -H "Content-Type: application/json" \
  -d '{"model": "default", "messages": [{"role": "user", "content": "Как настроить CI/CD?"}]}'
```

Альтернативно используйте заголовок `X-Auth-Token` (без префикса `Bearer`).

### Обновление токена

Токены доступа истекают через `ACCESS_TOKEN_MINUTES` (по умолчанию: 60 минут). Используйте refresh-токен для получения новой пары:

```bash
curl -X POST http://localhost:8080/v1/auth/refresh \
  -H "Content-Type: application/json" \
  -d '{"refresh_token": "dGhpcyBpcyBhIHJlZnJlc2g..."}'
```

Refresh-токены одноразовые — каждое обновление выдаёт новую пару.

### Выход из системы

Отозвать все токены текущего пользователя:

```bash
curl -X POST http://localhost:8080/v1/auth/logout \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1..."
```

## Интерфейс чата

### Базовый запрос

```bash
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "default",
    "messages": [{"role": "user", "content": "Как настроить GitLab CI?"}]
  }'
```

### RAG-специфичные параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `rag_version` | string | Запросить конкретную версию документа |
| `rag_force_refresh` | boolean | Обойти кеш ответов и повторно извлечь |

Расширения ответа (в заголовках `x-rag-*` или метаданных ответа):

| Расширение | Описание |
|-----------|----------|
| `rag_feedback_id` | Используйте этот ID для отправки отзыва на ответ |
| `rag_confidence` | Оценка достоверности (0.0–1.0) |
| `rag_sources` | Список использованных исходных документов |

### Стриминг

Поддерживается стандартный стриминг OpenAI:

```bash
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "default",
    "stream": true,
    "messages": [{"role": "user", "content": "Объясните пайплайн деплоя"}]
  }'
```

## Отправка обратной связи

Обратная связь — основной механизм повышения качества системы. Каждый ответ чата содержит `rag_feedback_id`, который используется для оценки ответа.

### Положительная обратная связь

Когда ответ точен и полезен:

```bash
curl -X POST http://localhost:8080/v1/feedback \
  -H "Content-Type: application/json" \
  -d '{
    "feedback_id": "fb_abc123def456",
    "rating": "positive"
  }'
```

### Отрицательная обратная связь

Когда ответ неточен или бесполезен:

```bash
curl -X POST http://localhost:8080/v1/feedback \
  -H "Content-Type: application/json" \
  -d '{
    "feedback_id": "fb_abc123def456",
    "rating": "negative",
    "comment": "Шаги CI/CD устарели для GitLab 16.x"
  }'
```

### Исправления

Эксперты могут предоставить исправленные ответы:

```bash
curl -X POST http://localhost:8080/v1/feedback \
  -H "Content-Type: application/json" \
  -d '{
    "feedback_id": "fb_abc123def456",
    "rating": "negative",
    "correction": "Для настройки GitLab CI в версии 16+ создайте .gitlab-ci.yml...",
    "comment": "Оригинальный ответ был для синтаксиса GitLab 14.x"
  }'
```

> **Примечание:** Для отправки отзывов требуется роль `expert` или `admin`.

## Как обратная связь улучшает результаты

Ваша обратная связь запускает непрерывный цикл улучшений:

### Положительная обратная связь → Обогащение знаний

Когда `ENRICHMENT_ENABLED=true`:

1. Получена положительная оценка ответа
2. Система извлекает пару вопрос-ответ (запрос + ответ)
3. Пара разбивается на чанки и векторизуется
4. Новый чанк индексируется в Qdrant

Это означает, что будущие запросы, похожие на оригинальный, найдут этот подтверждённый хороший ответ как кандидата для извлечения, улучшая полноту поиска по этой теме.

### Отрицательная обратная связь → Дообучение ранжировщика

Отрицательные оценки и исправления собираются как данные для обучения:

1. Функция `export_training_dataset()` извлекает все взаимодействия с исправлениями или положительной обратной связью
2. Они становятся парами (запрос, ответ) для дообучения ранжировщика
3. Исправления предпочитаются оригинальным ответам в обучающем наборе

Запустите экспорт вручную:

```python
from pathlib import Path
from proxy.app.core.hitl import export_training_dataset

export_training_dataset(Path("./training_data.jsonl"))
```

### Экспорт набора данных намерений

Логи взаимодействий также могут быть экспортированы как данные для обучения классификатора намерений:

```python
from pathlib import Path
from proxy.app.core.hitl import export_intent_dataset

export_intent_dataset(Path("./intent_data.jsonl"))
```

### Профиль пользователя → Персонализированное извлечение

Контекст пользователя (роли, группы, пространство имён) влияет на извлечение:

- **Изоляция пространства имён**: Пользователи в разных пространствах имён видят разные области документов
- **Уровень доступа**: Видимость документов фильтруется по уровню доступа пользователя
- **Группы**: Членство в группах может влиять на то, какие хранилища знаний запрашиваются

## Оценки достоверности

Каждый ответ включает оценку достоверности от 0.0 до 1.0. Эта оценка вычисляется путём комбинации нескольких сигналов:

### Компоненты оценки

| Сигнал | Вес | Описание |
|--------|-----|----------|
| Наличие контекста | Базовый | Штраф, если извлечённый контекст пуст или очень короткий |
| Соотношение контекста к ответу | 0.2 | Штраф, если контекст значительно короче ответа |
| Фразы неопределённости | 0.2 | Штраф за неуверенный язык («не уверен», «возможно») |
| Длина ответа | 0.15 | Штраф за очень короткие ответы |
| NLI-верификация | 0.4 | Сколько утверждений ответа подтверждается контекстом |

### Интерпретация оценок

| Диапазон | Значение | Рекомендуемое действие |
|----------|----------|----------------------|
| 0.8–1.0 | Высокая достоверность | Доверяйте ответу |
| 0.5–0.8 | Средняя достоверность | Проверьте важные детали |
| 0.0–0.5 | Низкая достоверность | Ответ может содержать галлюцинации; переформулируйте запрос или обратитесь к человеку |

### NLI-верификация

Когда NLI-верификация включена, система:

1. Разлагает ответ на атомарные утверждения
2. Проверяет каждое утверждение against извлечённый контекст
3. Сообщает, какие утверждения подтверждены, а какие нет

Это помогает обнаруживать галлюцинации — утверждения в ответе, не имеющие подтверждения в исходных документах.

### Качество извлечения

Система также оценивает качество извлечения с помощью классификатора в стиле CRAG:

- **Correct**: Чанки высоко релевантны запросу
- **Ambiguous**: Чанки частично релевантны
- **Incorrect**: Чанки не релевантны запросу

Если качество извлечения низкое, попробуйте переформулировать запрос более конкретно.

## Часто задаваемые вопросы

### В: Что происходит, когда аутентификация отключена?

Когда `AUTH_ENABLED=false` (по умолчанию), все эндпоинты доступны без аутентификации. Система работает в анонимном режиме с полным доступом. Это подходит для локальной разработки и внутренних развёртываний за VPN.

### В: Можно ли использовать систему без Keycloak?

Да. Система поддерживает два режима аутентификации:

1. **Локальный режим** (по умолчанию): Использует HS256 с `JWT_SECRET`. Регистрация и вход через `/v1/auth/register` и `/v1/auth/login`.
2. **Режим Keycloak/OIDC**: Когда установлен `KEYCLOAK_URL`, система автоматически находит конфигурацию OIDC и валидирует RS256-токены от Keycloak.

### В: Как получить роль expert?

По умолчанию новые регистрации получают роль `user`. Администратор должен повысить пользователей до `expert` через базу данных пользователей. В локальном режиме можно настроить начальных пользователей через переменную окружения `AUTH_VALID_USERS`.

### В: Мои отзывы хранятся постоянно?

Логи взаимодействий и обратная связь хранятся как JSONL-файлы в настроенном `LOG_DIR`. Файлы автоматически ротируются при превышении лимита размера (10 МБ для взаимодействий, 5 МБ для обратной связи), с сохранением до 5 резервных копий.

### В: Можно ли использовать виджет без аутентификации?

Да. Эндпоинт виджета (`/v1/widget`) и JavaScript (`/v1/widget.js`) являются публичными эндпоинтами, работающими без аутентификации. Они указаны как публичные в middleware аутентификации.

### В: Как система обрабатывает конфиденциальные данные?

- Пароли хешируются с помощью bcrypt (настраиваемое количество раундов)
- Refresh-токены хранятся как SHA-256 хеши
- Блокировка токенов предотвращает повторное использование отозванных токенов
- Токены доступа включают JTI (JWT ID) для индивидуального отзыва
- Все данные пользователей хранятся локально в SQLite — не требуются внешние сервисы

### В: Какие метрики доступны?

Когда `METRICS_ENABLED=true`, метрики Prometheus доступны по адресу `/metrics`, включая:

- Счётчики и латентности запросов
- Частота отправки отзывов
- Распределение оценок достоверности
- Попадания в кеш
