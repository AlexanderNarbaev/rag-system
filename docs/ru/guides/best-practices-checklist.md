**Последнее обновление:** 2026-06-24

## 5. Надёжность

| # | Критерий | Статус | Критерий прохождения | Проверка | Исправление |
|---|----------|--------|---------------------|----------|-------------|

| 5.2 | Повтор с backoff | ✅ Пройдено | `MAX_RETRIES=3`, задержка 1s/2s/4s; jitter | Убить LLM во время запроса → 3 повтора → ошибка | Реализовать экспоненциальный backoff с jitter |

| 5.4 | Multi-AZ / HA | ☐ Не пройдено | ≥ 2 реплики прокси, Qdrant, Neo4j, Redis; балансировщик; failover < 30s | Убить одну реплику → трафик на другую | Мигрировать с Docker Compose на Kubernetes |
| 5.5 | Автоматические бэкапы | ☐ Не пройдено | Qdrant снапшоты ежедневно, Neo4j дампы ежедневно, Redis RDB ежечасно; хранение вне хоста | `ls backup/` → свежие файлы бэкапов | Написать `scripts/backup.sh`; cron или K8s CronJob |
| 5.6 | Runbook аварийного восстановления | ☐ Не пройдено | Процедуры: восстановление из бэкапа, перестроение индексов, failover; RTO < 4ч, RPO < 1ч | Выполнить runbook в staging → система восстановлена за 4ч | Написать `docs/guides/disaster-recovery.md` |
| 5.7 | WAL-восстановление ETL | ✅ Пройдено | Чекпоинты после каждого источника; возобновление; `--reset-wal` | Запустить ETL, убить, перезапустить → продолжает с чекпоинта | Тестировать восстановление с повреждённым WAL |
| 5.8 | Грациозный старт/стоп | ✅ Пройдено | `lifespan` инициализирует кэши, закрывает соединения; запросы завершаются (до 30s) | `timeout 5 docker-compose stop proxy` → graceful shutdown | Добавить обработку SIGTERM/SIGINT |

| 5.10 | Dead letter queue | ☐ Не пройдено | Неудачные запросы в очередь; повтор 1мин/5мин/15мин (3 попытки) | Симуляция временного сбоя → запрос повторён → успех или постоянный сбой | Реализовать DLQ на Redis или in-memory |

## Как использовать этот чеклист

1. **Еженедельно:** Запускайте скрипт автоматической проверки. Расследуйте все сбои.
2. **При релизе:** Перед тегированием вручную проверьте все критерии с ✅ или 🟡.
3. **Приоритизация:** Сначала критические, затем высокие, затем средние.
4. **Отслеживание:** Создавайте GitHub issue для каждого критерия с ❌. Ссылайтесь на чеклист.
5. **Тренд баллов:** Отслеживайте общий балл (42.5/80). Цель: +5 баллов за минорную версию.

**Цель для v1.0:** ≥ 70/80 (87.5%) по всем измерениям.
