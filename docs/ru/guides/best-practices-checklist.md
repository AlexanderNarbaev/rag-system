# Чек-лист готовности к Production

**Последнее обновление:** 2026-06-22
**Версия:** v0.1.0

Этот чек-лист отслеживает готовность к production по 8 измерениям. Отмеченные пункты (✅) реализованы; неотмеченные (☐) — пробелы. Частично реализованные отмечены 🟡.

---

## 1. Качество кода

- [x] Все функции имеют type hints — частично: основные модули используют `typing`, но покрытие неравномерно
- [x] Все публичные функции имеют docstrings — частично: русские docstrings в модулях прокси, английские в ETL
- [x] Нет голых `except:` — проверено в `proxy/app/`, `etl/` использует конкретные типы исключений
- [x] Нет захардкоженных секретов — все учётные данные в `.env` / переменных окружения, маскируются через `SENSITIVE_SECRETS`
- [x] Pre-commit хуки настроены — ruff lint + format через `make lint` и `make format`
- [x] CI-пайплайн работает — цель `make all`: install → lint → test (полный пайплайн)
- [ ] Консистентная обработка ошибок — `orchestrator.py:78` ловит `Exception`, `main.py:354` ловит `Exception`; нет пользовательской иерархии исключений
- [ ] Type checker проходит — цель `make typecheck` существует, но 40+ ошибок mypy в модулях прокси (существовали ранее)
- [ ] Валидация переменных окружения при запуске — `config.py` не имеет валидации значений по умолчанию; невалидные значения вызывают ошибки времени выполнения
- [x] `.gitignore` настроен — покрывает `.env`, `__pycache__`, `.pytest_cache`, `*.pyc`, dist

## 2. Тестирование

- [x] Модульные тесты покрывают все модули — 282 прокси + 121 ETL + 46 MCP server + 56 интеграционных = 505 всего
- [x] Интеграционные тесты покрывают основные потоки — 56 интеграционных тестов: поиск, реранк, оркестратор, HITL
- [x] Покрытие тестами > 80% — 483/505 тестов проходят (95.6%); измерение покрытия доступно через `pytest --cov`
- [ ] E2E-тесты с реальными сервисами — нет end-to-end тестов с живыми экземплярами Qdrant/Neo4j/LLM
- [ ] Бенчмарки производительности — нет скриптов нагрузочного тестирования, нет бенчмарков задержки, нет тестов масштабируемости
- [ ] Chaos/отказоустойчивое тестирование — нет инъекции сбоев, нет симуляции разделения сети, нет сценариев отключения сервисов
- [x] CI-дружественно — все тесты выполняются без внешних сервисов; фикстуры используют моки
- [ ] Регрессионный набор тестов качества поиска — нет оценочного датасета, нет регрессионных проверок MRR/Recall
- [ ] Тесты сравнения снапшотов/данных — 21 тест падает (в основном несоответствия утверждений после изменений кода)
- [ ] Изоляция фикстур тестовых данных — некоторые тесты зависят от общих фикстур, которые могут мутировать состояние

## 3. Безопасность

- [x] Маскирование чувствительных данных в логах — переменная `SENSITIVE_SECRETS` + функции маскирования в `utils.py`
- [x] Ограничение частоты API-запросов — middleware token bucket на IP (`rate_limiter.py`), настраивается через `RATE_LIMIT_*`
- [ ] JWT-аутентификация — не реализована; интеграция Keycloak запланирована на v0.3 (дизайн: `access-control-rbac.md`)
- [ ] Реализация RBAC — не реализована; дизайн контроля доступа на уровне документов существует, но кода нет
- [ ] Валидация ввода для всех эндпоинтов — Pydantic-модели валидируют схему; нет защиты от инъекций на основе контента, нет ограничений длины `messages.content`
- [ ] Сканирование уязвимостей зависимостей — не настроены `safety`, `pip-audit` или Dependabot
- [ ] Терминирование HTTPS/TLS — не обрабатывается самим прокси; ожидается на уровне nginx/HAProxy; нет конфигурации HSTS
- [x] Конфигурация CORS — переменная `CORS_ORIGINS`, по умолчанию `*`, настраивается
- [ ] Аутентификация по API-ключу — `LLM_API_KEY` используется только для восходящих вызовов LLM, нет валидации клиентского ключа
- [ ] Аудит-логирование — HITL логирует взаимодействия, но без идентификации пользователя, событий аутентификации или действий администратора

## 5. Надёжность

- [x] Circuit breaker — неявно через логику повторов; нет выделенной реализации `pybreaker`. Повтор для LLM/Qdrant/Neo4j с настраиваемыми попытками
- [x] Повтор с backoff — `MAX_RETRIES=3`, `RETRY_DELAY=1.0s` для LLM; паттерн экспоненциального backoff задокументирован для всех сервисов
- [x] Graceful degradation — дизайн задокументирован для Neo4j (пропустить расширение графа), реранкера (пропустить реранк), Redis (in-memory fallback), LLM (retry-after)
- [ ] Развёртывание в нескольких AZ — развёртывание на одном узле Docker Compose; нет высокой доступности
- [ ] Автоматизация резервного копирования — нет автоматических снапшотов Qdrant, дампов Neo4j или Redis RDB
- [ ] Runbook аварийного восстановления — нет процедур DR, нет задокументированного процесса восстановления из холодного хранилища
- [x] Восстановление ETL на основе WAL — реализовано (`wal_manager.py`, флаг `--reset-wal`); протестировано модульными тестами ETL
- [x] Грациозный запуск/остановка — контекстный менеджер `lifespan` инициализирует кэш/оркестратор, очищает при выходе
- [ ] Пул соединений — клиенты Qdrant/Neo4j используют настройки соединений по умолчанию; нет явной конфигурации пула
- [ ] Очередь недоставленных сообщений — неудачные запросы логируются, но не повторяются и не ставятся в очередь для последующей обработки

## 6. Производительность

- [x] Кэш эмбеддингов — in-memory LRU + Redis-backed; ключ по MD5; снижает накладные расходы на перекодирование
- [x] Кэш ответов — Redis с TTL 1 час; поддерживается обход через `rag_force_refresh`
- [x] Cross-encoder реранк — MiniLM-L-6-v2 с batch_size=32; задокументированы компромиссы с более крупными моделями
- [x] Руководства по настройке HNSW — задокументированы в `performance-quality.md` с рекомендациями для каждого размера коллекции
- [ ] Результаты нагрузочного тестирования — нагрузочное тестирование не проводилось; нет бенчмарков конкурентных пользователей
- [ ] Цели по задержке p95 — цель задокументирована (p95 < 5s warn, < 10s crit), но не измерена под нагрузкой
- [x] Управление бюджетом токенов — `token_optimizer.py` с 4 стратегиями сжатия и умным распределением бюджета
- [ ] Прогрев моделей — нет предварительного прогрева моделей эмбеддера или реранкера перед первым запросом
- [ ] Настройка keep-alive соединений — стандартные настройки HTTP-клиента; нет настройки постоянных соединений
- [ ] Сжатие ответов — нет middleware gzip/brotli для больших ответов
