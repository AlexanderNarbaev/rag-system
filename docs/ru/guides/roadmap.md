## История версий

### v0.1.0 — Базовая инфраструктура (текущая)

**Выпущена:** Июнь 2026
**Статус:** ✅ Завершена

**Реализованные возможности:**
- Гибридный поиск (dense + sparse) с RRF-фьюжном через Qdrant
- Cross-encoder реранкинг с MiniLM-L-6-v2
- Контентно-адресуемое версионирование чанков (SHA-256) с горячим/холодным хранением
- Двухмодельная архитектура: SLM (лёгкая) для маршрутизации + LLM (полноразмерная) для генерации
- Агентный оркестратор LangGraph (7-узловой граф состояний, опционально)
- OpenAI-совместимый прокси-API (`/v1/chat/completions`, `/v1/models`, `/v1/health`)
- Многоуровневый кэш на Redis (эмбеддинги, реранк, ответы)
- Панель обратной связи Human-in-the-loop (Streamlit)
- Инкрементальный ETL на основе WAL с восстановлением по чекпоинтам
- GraphRAG через извлечение сущностей Neo4j и многошаговый обход
- Метрики Prometheus и структурированное логирование
- Ограничение частоты запросов (token bucket на IP)
- Дизайн graceful degradation для всех компонентов
- Поддержка автономного развёртывания (модели предварительно загружены)
- 919 тестов проходят (100% прохождения)
- 7 Architecture Decision Records, 4 C4-диаграммы, 8 руководств по проектированию

**Известные проблемы:**
- Нет оценочного датасета или метрик качества поиска
- Нет end-to-end тестов с реальными сервисами
- 40+ ошибок типов mypy в модулях прокси
- `USE_LANGGRAPH=false` по умолчанию; агентный путь опционален

### v0.2 — Самоулучшающийся RAG (текущая)

**Выпущена:** Июнь 2026
**Статус:** ✅ Завершена

**Тема:** Добавление оценки уверенности, активной обратной связи, VERIFY_CASCADE-маршрутизации и самообогащения базы знаний.

**Реализованные возможности:**
- **Оценка уверенности** — эвристическая оценка (достаточность контекста, соотношение контекст/ответ, обнаружение фраз неуверенности) с настраиваемым порогом
- **Активная обратная связь** — эндпоинт `/v1/feedback`, `rag_feedback_id` во всех ответах, сбор экспертных оценок
- **VERIFY_CASCADE-маршрутизация** — узел `check_confidence` в оркестраторе LangGraph; ответы с низкой уверенностью запускают цикл перезапроса (до `MAX_VERIFY_LOOPS`)
- **Самообогащение** — положительная обратная связь с исправлениями индексируется обратно в Qdrant как Q&A-пары (`ENRICHMENT_ENABLED`)
- **Оповещения администратора** — ответы с низкой уверенностью за пределами цикла вызывают оповещения (`ADMIN_ALERT_ENABLED`)
- **Метаданные ответа** — `rag_confidence` и `rag_feedback_id` внедряются в потоковые и не-потоковые ответы
- **919 тестов проходят** (100% прохождения)

### v0.3 — Оптимизация токенов и основы качества

**Цель:** Q3 2026
**Статус:** 🔵 Планирование

**Тема:** Снизить затраты токенов на 40% и установить измеримые базовые показатели качества.

#### Возможности

| # | Возможность | Описание | Критерии успеха |
|---|------------|----------|----------------|
| 1 | **Оценочный датасет поиска** | Создать 200+ размеченных пар запрос-документ из логов HITL + экспертная разметка | MRR, Recall@k, nDCG@k вычислимы для каждого коммита |
| 2 | **Автоматизированный конвейер оценки** | Скрипт `evaluate_retrieval.py`, вычисляющий MRR, Recall@20, nDCG@10, Precision@5 | Регрессионный тест в CI падает, если MRR < 0.75 |
| 3 | **Оценка заземления контекста** | Подключить проверку заземления на основе косинусного сходства в узел `generate` оркестратора | Оценка заземления логируется для каждого ответа; <0.4 вызывает "Я не знаю" |
| 4 | **Интеграция оптимизатора токенов** | Интегрировать `TokenOptimizer` в `build_context` (обрезка с учётом токенов) и основной конвейер | Использование токенов на запрос снижено на 25-40% |
| 5 | **Динамический top-k поиск** | SLM классифицирует сложность запроса → динамически корректирует `MAX_CHUNKS_RETRIEVAL` | На 30% меньше токенов для простых запросов |
| 6 | **Префиксное кэширование vLLM** | Включить `--enable-prefix-caching` для повторяющихся системных промптов | Устранение затрат токенов системного промпта после первого запроса в сессии |
| 7 | **Консистентная обработка ошибок** | Определить пользовательскую иерархию исключений (`RAGError`, `RetrievalError`, `LLMError` и др.) | Все блоки catch используют конкретные исключения |
| 8 | **Пробы готовности/живости** | Добавить эндпоинты `/v1/health/live` и `/v1/health/ready` | Docker/K8s health checks работают корректно |
| 9 | **Исправить 21 падающий тест** | Устранить несоответствия утверждений и проблемы зависимостей тестов | 505/505 тестов проходят (100%) |
| 10 | **Сканирование зависимостей** | Интегрировать `pip-audit` или `safety` в CI-пайплайн | CI падает при критических CVE |

#### Зависимости
- Логи HITL должны накопить достаточно данных взаимодействия для оценочного датасета
- Доступность экспертов для разметки (~10 часов)
- `token_optimizer.py` уже реализован; требуется подключение к конвейеру

#### Оценка усилий
- **Разработка:** 15-20 человеко-дней
- **Разметка:** 3-5 человеко-дней (доменный эксперт)
- **Всего:** ~4 недели при 1 FTE + эксперт на частичную занятость

### v0.4 — Безопасность и мультитенантность

**Цель:** Q4 2026
**Статус:** ⚪ Запланировано

**Тема:** Сделать систему безопасной для многопользовательского корпоративного развёртывания.

#### Возможности

| # | Возможность | Описание | Критерии успеха |
|---|------------|----------|----------------|
| 1 | **JWT-аутентификация** | Интеграция Keycloak для SSO; валидация JWT на всех эндпоинтах | Неаутентифицированные запросы возвращают 401 |
| 2 | **Реализация RBAC** | Ролевой доступ: admin, expert, user, read-only | Пользователи видят только авторизованные документы; эксперты могут отправлять обратную связь |
| 3 | **Изоляция данных пользователей** | Фильтрация Qdrant на пользователя/группу по пространству имён | Пользователь A не может получить приватные документы пользователя B |
| 4 | **Санитизация ввода** | Ограничения длины контента, защита от SQL/DQL-инъекций в параметрах запроса | Прохождение OWASP Top-10 инъекционных тестов |
| 5 | **Аудит-логирование** | Логирование аутентифицированных запросов с идентификацией пользователя, действием, временной меткой | Полный аудиторский след для комплаенса |
| 6 | **Терминирование HTTPS/TLS** | Задокументированная конфигурация обратного прокси nginx/HAProxy с TLS | Весь трафик шифруется при передаче |
| 7 | **Метрики качества чанкера** | Вычисление семантической когерентности, точности границ, коэффициента перекрытия во время ETL | Метрики логируются за каждый запуск ETL; алерт при когерентности < 0.70 |
| 8 | **Ротация логов** | Конфигурация logrotate для JSONL-логов HITL и логов прокси | Логи ротируются при 100MB, хранятся 7 дней |
| 9 | **A/B-тестовый харнесс** | Фреймворк для сравнения вариантов конвейера (LangGraph вкл/выкл, разные реранкеры) | Статистически значимое сравнение качества на < 500 запросах |
| 10 | **Покрытие интеграционных тестов для auth** | Тесты для всех сценариев RBAC | 100% путей аутентификации покрыто |

#### Зависимости
- Экземпляр Keycloak, развёрнутый в корпоративной среде
- Завершение оценочного датасета из v0.3 (необходим для A/B-тестирования)
- Одобрение сетевой/безопасностной команды для JWT-потока

#### Оценка усилий
- **Разработка:** 25-30 человеко-дней
- **Инфраструктура:** 5 человеко-дней (настройка Keycloak)
- **Всего:** ~6 недель при 1 FTE

### v0.5 — Мультимодальный RAG

**Цель:** Q1 2027
**Статус:** ⚪ Запланировано

**Тема:** Расширение за пределы текста на изображения, диаграммы и код.

#### Возможности

| # | Возможность | Описание | Критерии успеха |
|---|------------|----------|----------------|
| 1 | **Эмбеддинг и поиск изображений** | Интеграция CLIP/BLIP для индексации диаграмм и скриншотов | "Найди архитектурную диаграмму для сервиса X" возвращает правильное изображение |
| 2 | **Чанкинг с учётом кода** | AST-осведомлённое разделение для Python/JS/Java; чанки на уровне функций/классов | Поиск кода находит сигнатуры функций, а не только комментарии |
| 3 | **Извлечение таблиц** | Парсинг таблиц Confluence/Jira в структурированные представления | "Покажи таблицу бенчмарков производительности" возвращает табличные данные |
| 4 | **Мультимодальная сборка контекста** | Смешивание текста + изображений + кода в контексте LLM | LLM обрабатывает смешанный контент без путаницы |
| 5 | **Замыкание цикла обратной связи HITL** | Использование экспертных исправлений для файн-тюнинга реранкера на доменных данных | Улучшение MRR на ≥5% после 500 исправлений |
| 6 | **ColBERT late interaction** | Включение bge-m3 ColBERT multi-vectors для максимальной релевантности | Улучшение Recall@20 на ≥3% по сравнению с одним dense+sparse |
| 7 | **Автоматическая очистка холодного хранилища** | TTL-обрезка версий Parquet; хранить последние 5 версий на документ | Холодное хранилище остаётся в пределах 2× размера горячего хранилища |

#### Зависимости
- GPU с ≥24GB VRAM для инференса CLIP + ColBERT
- Минимум 500 экспертных исправлений, накопленных из HITL
- AST-парсеры для целевых языков

#### Оценка усилий
- **Разработка:** 30-35 человеко-дней
- **Оценка моделей:** 5 человеко-дней
- **Всего:** ~8 недель при 1 FTE

### v0.6 — Индексация в реальном времени и стриминг

**Цель:** Q2 2027
**Статус:** ⚪ Запланировано

**Тема:** Устранение задержки ETL с помощью потокового приёма данных.

#### Возможности

| # | Возможность | Описание | Критерии успеха |
|---|------------|----------|----------------|
| 1 | **Приём через вебхуки** | Вебхуки Confluence/GitLab запускают инкрементальное индексирование | Новый документ доступен для поиска в течение 30 секунд после публикации |
| 2 | **Потоковый ETL-конвейер** | Kafka/Redis Streams заменяют пакетный планировщик для обработки в реальном времени | Конвейер обрабатывает события в течение 5 секунд после поступления |
| 3 | **Живые upsert-ы Qdrant** | Атомарные обновления на уровне чанков без полной переиндексации | Нулевой простой при обновлении документов |
| 4 | **Потоковая генерация LLM** | Оптимизация SSE-стриминга для меньшего времени до первого токена | TTFT < 1s для кэшированных контекстов |
| 5 | **Эндпоинт прогрева моделей** | Предварительная загрузка эмбеддера/реранкера/LLM в память GPU при запуске | Задержка первого запроса равна последующим запросам |
| 6 | **Сжатие ответов** | Middleware gzip/brotli для больших ответов | Снижение размера тела ответа на 60%+ |

#### Зависимости
- Инфраструктура message broker (Kafka или Redis Streams)
- Конфигурация вебхуков на системах-источниках (Confluence, GitLab)
- Сетевая связность от систем-источников до ETL-машины

#### Оценка усилий
- **Разработка:** 25-30 человеко-дней
- **Инфраструктура:** 5-10 человеко-дней
- **Всего:** ~7 недель при 1 FTE
