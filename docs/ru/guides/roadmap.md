## История версий

### v0.1.0 — Базовая инфраструктура (текущая)

**Выпущена:** Июнь 2026
**Статус:** ✅ Завершена

**Реализованные возможности:**

- Гибридный поиск (dense + sparse) с RRF-фьюжном через Qdrant
- Cross-encoder реранкинг с MiniLM-L-6-v2
- Контентно-адресуемое версионирование чанков (SHA-256) с горячим/холодным хранением
- Двухмодельная архитектура: SLM (лёгкая) для маршрутизации + LLM (полноразмерная) для генерации
- Агентный оркестратор LangGraph (7-узловой граф состояний, опционально)
- OpenAI-совместимый прокси-API (`/v1/chat/completions`, `/v1/models`, `/v1/health`)
- Многоуровневый кэш на Redis (эмбеддинги, реранк, ответы)
- Панель обратной связи Human-in-the-loop (Streamlit)
- Инкрементальный ETL на основе WAL с восстановлением по чекпоинтам
- GraphRAG через извлечение сущностей Neo4j и многошаговый обход
- Метрики Prometheus и структурированное логирование
- Ограничение частоты запросов (token bucket на IP)
- Дизайн graceful degradation для всех компонентов
- Поддержка автономного развёртывания (модели предварительно загружены)
- 919 тестов проходят (100% прохождения)
- 7 Architecture Decision Records, 4 C4-диаграммы, 8 руководств по проектированию

**Известные проблемы:**

- Нет оценочного датасета или метрик качества поиска
- Нет end-to-end тестов с реальными сервисами
- 40+ ошибок типов mypy в модулях прокси
- `USE_LANGGRAPH=false` по умолчанию; агентный путь опционален

### v0.2 — Самоулучшающийся RAG (текущая)

**Выпущена:** Июнь 2026
**Статус:** ✅ Завершена

**Тема:** Добавление оценки уверенности, активной обратной связи, VERIFY_CASCADE-маршрутизации и самообогащения базы
знаний.

**Реализованные возможности:**

- **Оценка уверенности** — эвристическая оценка (достаточность контекста, соотношение контекст/ответ, обнаружение фраз
  неуверенности) с настраиваемым порогом
- **Активная обратная связь** — эндпоинт `/v1/feedback`, `rag_feedback_id` во всех ответах, сбор экспертных оценок
- **VERIFY_CASCADE-маршрутизация** — узел `check_confidence` в оркестраторе LangGraph; ответы с низкой уверенностью
  запускают цикл перезапроса (до `MAX_VERIFY_LOOPS`)
- **Самообогащение** — положительная обратная связь с исправлениями индексируется обратно в Qdrant как Q&A-пары (
  `ENRICHMENT_ENABLED`)
- **Оповещения администратора** — ответы с низкой уверенностью за пределами цикла вызывают оповещения (
  `ADMIN_ALERT_ENABLED`)
- **Метаданные ответа** — `rag_confidence` и `rag_feedback_id` внедряются в потоковые и не-потоковые ответы
- **919 тестов проходят** (100% прохождения)

### v0.6 — Индексация в реальном времени и стриминг

**Цель:** Q2 2027 (досрочно поставлен Q2 2026)
**Статус:** ✅ Завершено

**Тема:** Устранение задержки ETL с помощью потокового приёма данных.

#### Возможности

| # | Возможность                   | Описание                                                                                                                                                                                           | Критерии успеха                                                                                                            |
|---|-------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------|
| 1 | **Приём через вебхуки**       | Вебхуки Confluence/GitLab запускают инкрементальное индексирование через Redis Streams. Эндпоинт (`POST /webhook/confluence`) проверяет подписи и помещает события в очередь.                      | Новый документ доступен для поиска в течение 30 секунд после публикации. Верификация вебхуков проходит на всех источниках. |
| 2 | **Потоковый ETL-конвейер**    | Группы потребителей Redis Streams заменяют пакетный планировщик для обработки в реальном времени. Группы: `etl-extract`, `etl-chunk`, `etl-embed`, `etl-index`. WAL-чекпоинты после каждого этапа. | Конвейер обрабатывает события в течение 5 секунд после поступления. Consumer lag < 10 сообщений при нормальной нагрузке.   |
| 3 | **Живые upsert-ы Qdrant**     | Атомарные обновления на уровне чанков через `set_payload` и `upsert_points` без полной переиндексации. Сравнение хешей версий предотвращает избыточные записи.                                     | Нулевой простой при обновлении документов. Только изменённые чанки вызывают переиндексацию.                                |
| 4 | **Потоковая генерация LLM**   | Оптимизация SSE-стриминга: пул соединений, chunked transfer encoding, уменьшенная начальная буферизация. TTFT измеряется через гистограмму Prometheus.                                             | TTFT < 1s для кэшированных контекстов. TTFT < 3s для некэшированного поиска.                                               |
| 5 | **Эндпоинт прогрева моделей** | `POST /v1/admin/warmup` предзагружает эмбеддер, реранкер и SLM в память GPU/CPU. Health check проверяет завершение прогрева. Прогрев при старте через lifespan.                                    | Задержка первого запроса равна последующим (±100ms). Прогрев завершается в течение 30s.                                    |
| 6 | **Сжатие ответов**            | Middleware gzip/brotli через Starlette `GZipMiddleware`. Сжимает ответы > 1KB. Настраивается через `COMPRESSION_ENABLED` и `COMPRESSION_LEVEL`.                                                    | Снижение размера тела ответа на 60%+ для JSON/текста. < 5ms накладных расходов CPU на запрос.                              |

#### Детали реализации

**Архитектура Redis Streams:**

- Поток: `etl:events` с группами потребителей `etl-extract`, `etl-chunk`, `etl-embed`, `etl-index`
- Каждая группа обрабатывает события независимо с WAL-чекпоинтами
- Ожидающие сообщения мониторятся через `XPENDING` для обнаружения consumer lag
- Dead letter stream `etl:events:dlq` для неудачных событий (макс. 3 повтора)
- Конфигурация: `STREAMING_ETL_ENABLED=true`, `REDIS_STREAMS_URL`

**Конфигурация вебхуков:**

- Confluence: `POST /webhook/confluence` с проверкой HMAC-SHA256 подписи
- GitLab: `POST /webhook/gitlab` с валидацией заголовка `X-Gitlab-Token`
- Поддерживаемые события: `page_created`, `page_updated`, `page_removed` (Confluence); `push`, `merge_request_merge` (
  GitLab)
- Секрет вебхуков настраивается через `WEBHOOK_SECRET`

**Прогрев моделей:**

- `POST /v1/admin/warmup` запускает фиктивный инференс на эмбеддере, реранкере и SLM
- Опционально: прогрев LLM через `WARMUP_LLM=true` (отправляет однотокеновое завершение)
- Kubernetes: post-start hook вызывает прогрев перед маркировкой пода как ready
- Prometheus-метрика: `rag_warmup_completed` gauge (0/1)

**Бенчмарки сжатия:**

- JSON ответы (rag_sources, chat completions): снижение на 65-72%
- SSE стрим-чанки: не сжимаются (transfer-encoding: chunked)
- HTML ответы (health dashboard): снижение на 75-80%
- Brotli даёт ~5% лучшее сжатие чем gzip ценой +10ms CPU

#### Зависимости

- Redis Streams (требуется Redis 5.0+)
- Конфигурация вебхуков на системах-источниках (Confluence, GitLab)
- Сетевая связность от систем-источников до ETL-машины

#### Оценка усилий

- **Разработка:** 25-30 человеко-дней
- **Инфраструктура:** 5-10 человеко-дней
- **Всего:** ~7 недель при 1 FTE

**Тема:** Достижение RAG Level 5 с полной самокоррекцией, агентными инструментами и мультиязычной поддержкой.

#### Возможности

| #  | Возможность                               | Описание                                                                                                                                                    | Критерии успеха                                                                            |
|----|-------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------|--------------------------------------------------------------------------------------------|
| 1  | **HyDE-расширение запросов**              | Генерация гипотетических документов из запросов для улучшения разреженного поиска                                                                           | Улучшение Recall на 10%+ для технических запросов с редкой терминологией                   |
| 2  | **CRAG-оценщик**                          | Многофакторная оценка качества поиска: распределение score (0.4), коэффициент покрытия (0.3), фактор количества результатов (0.2), декей актуальности (0.1) | Оценка уверенности корректно определяет действие: USE, REWRITE, EXPAND, FALLBACK           |
| 3  | **Циклы саморефлексии**                   | Шаг пост-генерационной критики: LLM перечитывает свой ответ с учётом найденного контекста, выявляет несоответствия                                          | Оценка саморефлексии коррелирует с экспертной обратной связью с r > 0.7                    |
| 4  | **Обнаружение и заземление галлюцинаций** | NLI-верификация ответов: проверка косинусного сходства эмбеддингов + классификация entailment                                                               | Уровень галлюцинаций < 5% для всех типов запросов                                          |
| 5  | **Корректирующая ре-генерация**           | Ответы с низкой уверенностью запускают ре-генерацию с расширенным контекстом, промптом на фактическую точность или другой температурой                      | 90% изначально низко-уверенных ответов улучшаются до приемлемого уровня после ре-генерации |
| 6  | **Агентный вызов инструментов**           | Живые запросы к API Confluence/Jira/GitLab через function calling; получение данных в реальном времени помимо индексированных знаний                        | Успешность вызова инструментов > 95%; задержка ответа < 3s для запросов с инструментами    |
| 7  | **Мультиязычная поддержка**               | Полная интернационализация: генерация ответов на RU, EN, DE, FR, ZH; параметр `lang` в чат-завершениях; кросс-языковые бенчмарки                            | Кросс-языковой MRR > 0.75 для всех поддерживаемых языков                                   |
| 8  | **Кросс-языковые бенчмарки**              | Оценочный датасет с мультиязычными парами запрос-документ; MRR, Recall@20, nDCG@10 для каждого языка                                                        | Показатели по языкам соответствуют порогам качества (> 0.80 MRR)                           |
| 9  | **Коннекторы к живым источникам**         | Прямая интеграция с Confluence REST API, Jira REST API, GitLab API для доступа к данным в реальном времени наряду с индексированными данными                | Комбинация живых источников с индексированным поиском улучшает свежесть ответов на 40%+    |
| 10 | **Графовые паттерны саморефлексии**       | Обогащение графа знаний Neo4j паттернами саморефлексии: рёбра валидации ответ-чанк, оценки уверенности связывания сущностей                                 | Граф-усиленная саморефлексия снижает ложные флаги галлюцинаций на 30%                      |
| 11 | **LLMLingua сжатие**                      | Сжатие промптов на уровне токенов для оптимизации контекста в длинных документах                                                                            | 2-5x коэффициент сжатия с потерей информации < 5%                                          |
| 12 | **LongContextReorder**                    | Переранжирование документов с значимым контентом по краям (начало/конец) для борьбы с эффектом "lost in the middle"                                         | Улучшение nDCG на 5%+ для документов с длинным контекстом                                  |

#### Зависимости

- Развёрнута NLI-модель для классификации entailment
- Доступна HyDE-модель (та же LLM или SLM) для генерации гипотетических документов
- Собран кросс-языковой оценочный датасет
- API-эндпоинты Confluence/Jira/GitLab доступны с прокси

#### Оценка усилий

- **Разработка:** 35-45 человеко-дней
- **Оценка:** 5-10 человеко-дней
- **Всего:** ~10 недель при 1 FTE
