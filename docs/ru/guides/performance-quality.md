# Производительность и качество: лучшие практики

## 1. Производительность поиска

### 1.1 Настройка Qdrant HNSW

Индекс HNSW определяет компромисс между скоростью поиска и полнотой. Настройка в зависимости от размера коллекции:

| Размер коллекции | ef_construct | ef_search | m | Ожидаемый Recall@10 |
|---|---|---|---|---|
| <100K векторов | 128 | 64 | 16 | >0.98 |
| 100K–1M векторов | 200 | 128 | 24 | >0.96 |
| >1M векторов | 256 | 200 | 32 | >0.94 |

Установка при создании коллекции Qdrant:
```python
models.HnswConfigDiff(ef_construct=200, m=24, ef=128)
```

**Режим на диске** для разреженных векторов: Установите `on_disk_payload=True` и `on_disk=True` в конфигурации коллекции. Для разреженных векторов bge-m3 (до 250K измерений на вектор) индексация на диске снижает использование RAM на 60% с увеличением задержки всего на ~5%.

### 1.2 Стратегии квантизации

| Стратегия | Снижение RAM | Влияние на Recall | Когда использовать |
|---|---|---|---|
| Scalar (int8) | 4× | <1% | По умолчанию для production, всегда включать |
| Product (PQ) | 16× | 2–4% | Для >5M векторов, ограниченный бюджет RAM |
| Binary (BQ) | 32× | 5–8% | Для >10M векторов, поиск предварительно фильтруется |

Включить скалярную квантизацию по умолчанию:
```python
models.ScalarQuantization(scalar=models.ScalarQuantizationConfig(type=models.ScalarType.INT8))
```

### 1.3 Пакетная обработка запросов

Для сценариев с несколькими запросами (параллельные запросы пользователей, декомпозиция запросов) группируйте плотные эмбеддинги:

```python
vectors = embedder.encode(queries, batch_size=32, normalize_embeddings=True)
```
Используйте `batch_size=32` для GPU (максимальная пропускная способность), `batch_size=8` для CPU (меньшее давление на память).

### 1.4 Оптимизация коэффициента попадания в кэш

Текущий дизайн ключа кэша в `retrieval.py:89` использует `md5(text.encode()).hexdigest()`. Улучшения:

| Уровень кэша | Шаблон ключа | TTL | Ожидаемый Hit Rate |
|---|---|---|---|
| Эмбеддинг | `embed:{md5(text)}` | 1 час | 15–25% (помогает переписывание запросов) |
| Результат поиска | `search:{md5(query)}:{version}:{top_k}` | 5 мин | 5–10% |
| Ответ LLM | `llm:{md5(prompt)}` | 10 мин | 3–8% (семантическая дедупликация) |
| Оценки реранка | `rerank:{md5(query)}:{md5(chunk_ids)}` | 5 мин | 8–12% |

Рекомендуемая конфигурация Redis: `maxmemory 2GB`, `maxmemory-policy allkeys-lru`.

## 2. Оптимизация инференса

### 2.1 Пакетная обработка эмбеддингов

Размер батча `SentenceTransformer.encode()`:
- **GPU (24GB VRAM)**: `batch_size=64` для bge-m3, 9ms на батч
- **CPU (32 ядра)**: `batch_size=16`, распараллеливание через процессы `pool`
- **Смешанный**: GPU для плотных, CPU для разреженных (разреженное кодирование bge-m3 привязано к CPU)

Включите `show_progress_bar=False` в production для избежания накладных расходов tqdm.

### 2.2 Компромисс реранкера

Текущий реранкер: `cross-encoder/ms-marco-MiniLM-L-6-v2` (80M параметров, `proxy/app/config.py:25`).

| Модель | Задержка на пару | Улучшение MRR | VRAM |
|---|---|---|---|
| MiniLM-L-6-v2 (текущий) | 8ms | +15% над dense | 0.5GB |
| MiniLM-L-12-v2 | 15ms | +18% | 1GB |
| bge-reranker-v2-m3 | 25ms | +22% | 2GB |

**Рекомендация**: Оставайтесь на MiniLM-L-6-v2 для <50 чанков. Переключайтесь на bge-reranker-v2-m3 только если MRR падает ниже 0.75 в мониторинге. Группируйте входы реранкера с `batch_size=32`.

### 2.3 Инференс LLM

Ваша настроенная LLM через адаптер бэкенда с оптимизациями:

| Параметр | Значение | Обоснование |
|---|---|---|
| `--max-model-len` | 131072 | Соответствует максимальному контексту модели |
| `--gpu-memory-utilization` | 0.92 | Оставить 8% для пиков KV-кэша |
| `--enable-prefix-caching` | true | Переиспользование KV-состояния системного промпта |
| `--max-num-seqs` | 4 | Конкурентные запросы для RAG-прокси |
| `--dtype` | awq (4-bit) | Квантизация AWQ значительно снижает VRAM |

Квантизация AWQ: 4-bit с размером группы 128, сохраняет перплексию в пределах 1% от FP16. Для GPU с <16GB VRAM — обязательно.

### 2.4 Маршрутизация SLM vs LLM

Двухмодельная архитектура маршрутизирует запросы:
- **SLM (ваша лёгкая модель)**: Переписывание запросов, извлечение сущностей (быстрый путь, <50ms)
- **LLM (ваша полноразмерная модель)**: Генерация финального ответа (медленный путь, 2–8s)

Эвристика маршрутизации: если `llm_router` обнаруживает запрос, требующий только поиска фактов (без рассуждений), ответить прямо из найденного контекста без вызова LLM — экономия 2–3s задержки.

### 2.5 Размер потокового чанка

Для ответов с `stream=True` выводите чанки по **16–32 токена** (по умолчанию для большинства бэкендов — 16). Это обеспечивает плавный UX без чрезмерных накладных расходов HTTP-фреймов.

## 3. Управление памятью

### 3.1 Выгрузка моделей

| Модель | GPU VRAM | CPU RAM | Стратегия |
|---|---|---|---|
| bge-m3 (эмбеддер) | 2.2GB | 0 | Держать на GPU постоянно |
| MiniLM-L-6-v2 (реранкер) | 0.5GB | 0 | Держать на GPU |
| Ваша LLM (квантизованная) | варьируется | 0 | Только GPU, управляется бэкендом |
| Ваша SLM (лёгкая) | ~1.8GB | 0 | GPU при наличии VRAM, иначе CPU |
| spaCy ru_core_news_sm | 0 | 50MB | Только CPU |

Общий бюджет GPU VRAM: варьируется в зависимости от выбора модели. При использовании карты 16GB переместите SLM на CPU и уменьшите `gpu-memory-utilization` до 0.88.

### 3.2 Размер кэша эмбеддингов

In-memory кэш эмбеддингов (`retrieval.py:88-95`): при 100K документах и 10 чанках каждый — ожидается ~500K уникальных эмбеддингов запросов+чанков. При 1024 float на плотный вектор × 4 байта = 4KB каждый → **2GB** для кэша плотных эмбеддингов. Установите Redis `maxmemory 2GB` с вытеснением на основе TTL для соблюдения бюджета.

### 3.3 Политика памяти Redis

```
maxmemory 2gb
maxmemory-policy allkeys-lru
```
`allkeys-lru` обеспечивает высокий hit rate для эмбеддингов и результатов поиска. Мониторинг через `INFO stats` — цель: `evicted_keys` около нуля. Если вытеснений >100/час, увеличьте `maxmemory` до 4GB или уменьшите TTL эмбеддингов с 3600s до 1800s.

### 3.4 Профилирование памяти ETL

Горячие точки памяти Python ETL-пайплайна:
- **spaCy**: `nlp.pipe(texts, batch_size=50)` вместо последовательного `nlp(text)` — снижение памяти на 40%
- **SentenceTransformer**: Явно вызывать `torch.cuda.empty_cache()` после каждого батча из 100 чанков
- **Загрузка Neo4j**: Использовать `UNWIND $batch` с `batch_size=500` (уже реализовано в `neo4j_loader.py:34`)

Профилирование с `memory_profiler`:
```python
@profile
def extract_batch(self, chunks): ...
```

## 4. Качество конвейера

### 4.1 Метрики качества чанкера

Оценка семантического чанкера (`etl/chunker/semantic_chunker.py`) по:

| Метрика | Цель | Измерение |
|---|---|---|
| Семантическая когерентность (косинусное сходство внутри чанка) | >0.75 | Среднее косинусное сходство предложений внутри чанка |
| Точность границ | >0.85 | % границ чанков на разрывах разделов/заголовков |
| Коэффициент перекрытия | 10–15% | Токенов, общих между последовательными чанками |
| Размер чанка (токенов) | 256–1024 | P50: 512, P95: 900 |

### 4.2 Качество поиска (офлайн-оценка)

Запускать еженедельно на размеченном оценочном наборе из 200+ пар запрос-документ:

| Метрика | Базовый (только dense) | Цель (hybrid+rerank) |
|---|---|---|
| MRR (Mean Reciprocal Rank) | 0.62 | >0.80 |
| nDCG@10 | 0.55 | >0.75 |
| Recall@20 | 0.78 | >0.90 |
| Recall@50 | 0.85 | >0.95 |

### 4.3 Влияние реранкера

Измерьте дельту: `MRR_rerank − MRR_dense`. Если дельта < 0.10, реранкер не добавляет достаточно ценности — выясните, возвращает ли поиск уже почти идеальный ранжинг (маловероятно) или модель реранкера не соответствует домену. Рассмотрите файн-тюнинг на доменных суждениях релевантности из данных HITL.

### 4.4 Обнаружение галлюцинаций

Реализовать оценку заземления контекста:
```python
grounding_score = cos_sim(embed(answer), embed(context))
```
- Score > 0.7: Хорошо заземлено
- Score 0.4–0.7: Частичное заземление, пометить для проверки HITL
- Score < 0.4: Вероятная галлюцинация, вернуть "У меня недостаточно информации"

Логировать оценку заземления с каждой генерацией для мониторинга.

### 4.5 Достаточность поиска

Узел `check_sufficiency` в `orchestrator.py:127-146` использует простую эвристику (`avg_score < 0.6`). Модернизация до оценки на основе LLM:

```python
sufficiency_prompt = f"""
Query: {query}
Retrieved context summary (10 chunks):
{[c['text'][:200] for c in chunks]}

Does this context contain enough information to answer the query?
Answer YES or NO with confidence 0-100.
"""
```
Если уверенность < 70, запустить расширение графа. Если < 50, запустить цикл переписывания запроса (ограничено `MAX_RETRIEVAL_LOOPS=3`).

## 6. Масштабируемость

### 6.1 Горизонтальное масштабирование

Слой прокси stateless (состояние сессии в Redis, чекпоинтер LangGraph в MemorySaver/RedisSaver). Масштабирование:
- **Docker Compose**: `docker-compose up -d --scale rag-proxy=3`
- **Kubernetes**: HPA при CPU >70% или задержке запросов p95 >3s
- **Балансировка нагрузки**: Round-robin на уровне nginx/HAProxy, не требуются sticky sessions

### 6.2 Шардирование Qdrant

Для коллекций >10M векторов:
```python
models.CreateCollection(
    shard_number=4,
    replication_factor=2,
    ...
)
```
- **Количество шардов**: 4 шарда для 10M–50M векторов, 8 шардов для >50M
- **Фактор репликации**: 2 (допускает отказ 1 узла)
- Каждый шард: ~2.5M векторов при 1024-dim = ~10GB RAM (со скалярной квантизацией)

### 6.3 Распараллеливание ETL

Параллелизм на уровне источников в `scheduler/run_etl.py`:
- **Confluence**: 3 воркера (ограничено API, 10 req/s)
- **Jira**: 5 воркеров (параллелизм на уровне задач)
- **GitLab**: 3 воркера (уровень коммитов + уровень файлов)

Используйте `concurrent.futures.ProcessPoolExecutor(max_workers=8)` для этапов чанкинга/эмбеддинга. Выделите GPU-ускоренный эмбеддинг в отдельный пул воркеров для избежания CUDA OOM.

### 6.4 Тир-хранение холодных данных

`LiveVectorLake` (`etl/indexer/live_vector_lake.py`): Хранить текущую версию (+1 предыдущую) в Qdrant (горячее). Перемещать версии старше 2 эпох в файлы Parquet на диске с отслеживанием манифеста. Запросы с параметром `version=` обращаются к холодному хранилищу (задержка 50–200ms) с прозрачным кэшированием.

## 7. Отказоустойчивость

### 7.1 Логика повторов

Экспоненциальный backoff с jitter:

| Сервис | Макс. повторов | Базовая задержка | Макс. задержка | Jitter |
|---|---|---|---|---|
| LLM backend | 3 | 1s | 30s | ±20% |
| Qdrant | 5 | 100ms | 5s | ±50% |
| Neo4j | 3 | 500ms | 10s | ±25% |
| Эмбеддер (локальный) | 0 | — | — | Нет сети, нет повтора |

Уже реализовано в `neo4j_loader.py:83` для Neo4j. Расширить паттерн на вызовы LLM и Qdrant.

### 7.2 Паттерн Circuit Breaker

Защита от каскадных отказов:
- **Circuit Qdrant**: Открыть после 5 последовательных ошибок в окне 30s. Полуоткрыть через 60s, протестировать 1 пробным запросом.
- **Circuit Neo4j**: Открыть после 3 ошибок в окне 20s. Полуоткрыть через 45s.
- **Circuit LLM**: Открыть после 3 таймаутов в окне 60s. Полуоткрыть через 90s.

Использовать библиотеку `pybreaker` или ручную реализацию с флагами `asyncio.Event`.

### 7.3 Graceful Degradation

| Отказ компонента | Режим деградации | Влияние на пользователя |
|---|---|---|
| Neo4j недоступен | Пропустить graph_expand, выдать только векторный контекст | Чуть менее связные ответы |
| Reranker OOM/ошибка | Пропустить реранк, использовать сырые гибридные оценки | Top-5 чанков чуть менее точны |
| Redis недоступен | Fallback к InMemoryCache | Пик задержки холодного старта (~100ms→500ms) |
| Таймаут LLM | Вернуть "Сервис временно деградирован, повторите через 30s" | Пользователь повторяет |
| Qdrant недоступен | Вернуть 503 с заголовком Retry-After | Полный отказ |

Прокси никогда не должен падать при отказе компонента — всегда возвращать лучший доступный ответ.

### 7.4 Восстановление ETL на основе WAL

ETL-пайплайн использует Write-Ahead Log в `etl/scheduler/wal/`:
- Перед каждым шагом извлечения записывать `{source}_{timestamp}_{checkpoint}.wal` с источником, информацией о батче, прогрессом
- При сбое/перезапуске воспроизвести WAL для поиска последнего зафиксированного чекпоинта
- Возобновить извлечение с `last_processed_id` в чекпоинте
- Очищать записи WAL старше 7 дней

Реализован как `CheckpointManager` в планировщике с:
```python
wal_format = {
    "source": "confluence",
    "batch_id": 42,
    "last_processed_id": "page_12345",
    "entities_loaded": 15600,
    "timestamp": "2026-06-22T10:30:00Z"
}
```
