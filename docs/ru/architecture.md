# Архитектура

**Версия:** v2.0.0 | **Обновлено:** 2026-07-10

Этот документ описывает архитектуру системы RAG Knowledge Assistant — OpenAI-совместимого прокси с полным ETL-пайплайном
для корпоративного поиска знаний.

---

## 1. Обзор системы

Система состоит из двух основных компонентов, работающих на отдельных серверах:

| Компонент        | Назначение                                                                       | Порт           |
|------------------|----------------------------------------------------------------------------------|----------------|
| **ETL Pipeline** | Извлечение данных, чанкинг, эмбеддинг, индексация в векторные/графовые хранилища | Н/A (пакетный) |
| **RAG Proxy**    | OpenAI-совместимый API с гибридным поиском, реранкингом и генерацией через LLM   | `8080`         |

Эти компоненты используют общие хранилища Qdrant и Neo4j, но работают независимо. ETL-пайплайн заполняет базы данных;
прокси опрашивает их при обслуживании запросов.

### Вспомогательные сервисы

| Сервис             | Назначение                                                  | Опционально |
|--------------------|-------------------------------------------------------------|-------------|
| **Redis**          | Многоуровневый кэш (эмбеддинги, результаты реранка, ответы) | Да          |
| **Neo4j**          | Граф знаний для контекстного расширения на основе сущностей | Да          |
| **HITL Dashboard** | Streamlit-интерфейс экспертной проверки                     | Да          |
| **MCP Server**     | Model Context Protocol сервер для интеграции с IDE          | Да          |
| **MLflow + MinIO** | Отслеживание экспериментов и хранение артефактов моделей    | Да          |

## 3. Потоки данных

### 3.1 Поток ETL-пайплайна

```
Источники данных → Извлечь → Нарезать → Вложить → Индексировать
     │                                                  │
     │                                                  ▼
     │                                           ┌──────────┐
     └── Извлечение сущностей ──────────────────│  Qdrant  │
                                                │  Neo4j   │
                                                └──────────┘
```

1. **Извлечение**: Коннекторы извлекают сырые данные из Confluence, Jira, GitLab, локальных файлов (документы, книги,
   логи чатов).
2. **Нарезка**: Семантический чанкер разбивает документы на перекрывающиеся сегменты с SHA-256 адресацией содержимого
   для дедупликации.
3. **Эмбеддинг**: BAAI/bge-m3 генерирует плотные (1024-dim) и разреженные векторы для каждого чанка.
4. **Индексация**: Векторы сохраняются в Qdrant с гибридным поиском (dense + sparse + ColBERT). Сущности извлекаются
   через spaCy NER и загружаются в Neo4j.
5. **WAL**: Журнал предзаписи отслеживает состояние обработки для инкрементальных обновлений и восстановления после
   сбоев.

### 3.2 Поток запросов

```
Клиент → API → [Auth] → [Проверка кэша] → Гибридный поиск → Контроль доступа
                                                                    │
       ┌────────────────────────────────────────────────────────────┘
       ▼
  Реранкер → Дедуп → Сборщик контекста → Бюджет токенов → LLM → Ответ
       │                                                        │
       │                                                        ▼
       │                                              ┌──────────────┐
       └──────────────────────────────────────────────│  Уверенность │
                                                      │  + Feedback  │
                                                      └──────────────┘
```

1. **Запрос**: Клиент отправляет OpenAI-совместимый запрос на chat completion.
2. **Аутентификация**: Проверка JWT и роли RBAC (если `AUTH_ENABLED=true`).
3. **Кэш**: Проверка кэша Redis/in-memory на наличие предыдущего ответа (если `rag_force_refresh` не установлен).
4. **Гибридный поиск**: Qdrant выполняет dense + sparse + ColBERT поиск с RRF фьюжн.
5. **Контроль доступа**: Пост-ретривальная фильтрация на основе ролей пользователя и ACL документов.
6. **Реранкинг**: Cross-encoder (MiniLM-L-6-v2) оценивает и фильтрует top-50 до top-20 чанков.
7. **Дедупликация**: Дедуп на основе SHA-256 хеша содержимого, фильтрация по версиям.
8. **Сборка контекста**: Конструкция контекста с бюджетом токенов и сжатием.
9. **Генерация LLM**: Адаптер провайдера маршрутизирует к настроенному бэкенду (vLLM, llama.cpp и т.д.).
10. **Уверенность**: Эвристическая + опциональная NLI-оценка.
11. **Ответ**: OpenAI-совместимый JSON с RAG-расширениями (`rag_feedback_id`, `rag_confidence`, `rag_sources`).

### 3.3 Агентный поток (при `USE_LANGGRAPH=true`)

Когда включён оркестратор LangGraph, запрос проходит через 10-узловой граф состояний:

```
rewrite → retrieve → check → rerank → graph_expand
   → build → generate → call_tools → reflect → check
```

- **rewrite**: Декомпозиция запроса через SLM на подзапросы
- **retrieve**: Многошаговый поиск с настраиваемыми циклами
- **check**: CRAG-оценщик оценивает качество поиска
- **graph_expand**: Многошаговый обход сущностей Neo4j
- **generate**: Генерация ответа LLM
- **call_tools**: Вызов зарегистрированных инструментов (запросы к Confluence/Jira/GitLab в реальном времени)
- **reflect**: Самокритика и детекция галлюцинаций
- **check**: Финальная проверка уверенности

## 4. Описание компонентов

### 4.1 ETL Pipeline

ETL-пайплайн работает независимо, обычно на отдельном сервере от прокси.

#### Экстракторы

| Экстрактор           | Источник                             | Выход                          |
|----------------------|--------------------------------------|--------------------------------|
| `confluence.py`      | REST API Confluence                  | Страницы, вложения, метаданные |
| `jira.py`            | REST API Jira                        | Задачи, комментарии, переходы  |
| `gitlab.py`          | REST API GitLab                      | Репозитории, MR, задачи, вики  |
| `doc_extractor.py`   | Локальные файлы (PDF, DOCX, MD, TXT) | Текст документа + метаданные   |
| `book_extractor.py`  | Электронные книги (EPUB, FB2)        | Чанки по главам                |
| `chat_extractor.py`  | Экспорт истории чатов                | Цепочки разговоров             |
| `image_extractor.py` | Изображения в документах             | Текст OCR                      |

Все экстракторы наследуют `base_extractor.py`, который предоставляет:

- Инкрементальное извлечение через хеширование содержимого
- Обработку ошибок и логику повторных попыток
- Нормализацию метаданных

#### Чанкер

- **`semantic_chunker.py`**: Разбивает документы по семантическим границам (абзацы, секции) с настраиваемым перекрытием.
- **`hash_versioning.py`**: SHA-256 контентно-адресуемые чанки для дедупликации и отслеживания версий.
- **`code_chunker.py`**: AST-aware нарезка для исходного кода (Python, JavaScript, Java).
- **`table_extractor.py`**: Структурированное извлечение таблиц из документов.

#### Индексатор

- **`qdrant_hybrid.py`**: Создаёт и управляет коллекциями Qdrant с плотными, разреженными и ColBERT векторными
  индексами. Реализует RRF фьюжн для гибридного поиска.
- **`live_vector_lake.py`**: Загрузка векторов в реальном времени для потоковых источников данных.
- **`wal_manager.py`**: Журнал предзаписи для восстановления после сбоев и инкрементальной обработки. Отслеживает
  состояния чанков (pending, processing, indexed, failed).

#### Построитель графа

- **`entity_extractor.py`**: Извлечение сущностей на основе spaCy NER — 10 типов: Person, Document, Project, Component,
  Technology, Team, Meeting, Decision, Milestone, Issue.
- **`neo4j_loader.py`**: Загружает сущности и 9 типов связей в Neo4j. Поддерживает инкрементальные обновления.
- **`schema.yaml`**: Определяет схему графа (типы сущностей, типы связей, ограничения).

#### Планировщик

- **`run_etl.py`**: Главный оркестратор, запускающий полный пайплайн (extract → chunk → embed → index → graph).
- **`stream_producer.py` / `stream_consumer.py`**: Kafka-based потоковый ETL для источников данных в реальном времени.
- **`webhook_server.py`**: HTTP webhook-приёмник для push-ориентированного получения данных.
- **`cold_storage_cleanup.py`**: Управление хранением Parquet в холодном хранилище.

### 4.2 RAG Proxy

Прокси — это FastAPI-приложение, обслуживающее запросы на порту `8080`.

#### Уровень API (`main.py`)

- OpenAI-совместимый эндпоинт `/v1/chat/completions` (потоковый и без потока)
- 25+ эндпоинтов: чат, модели, здоровье, аутентификация, обратная связь, инструменты, админ, метрики
- Модели запросов/ответов Pydantic с валидацией
- Стек middleware: CORS → Auth → Correlation ID → Request ID → Logging → Rate Limit → Compression

#### Поиск (`retrieval.py`)

- Гибридный поиск, объединяющий dense, sparse и ColBERT векторы
- Reciprocal Rank Fusion (RRF) для нормализации оценок
- Настраиваемый `MAX_CHUNKS_RETRIEVAL` (по умолчанию: 50)
- Graceful degradation: возвращает пустые результаты при сбое Qdrant

#### Реранкер (`rerank.py`)

- Cross-encoder модель (по умолчанию: `cross-encoder/ms-marco-MiniLM-L-6-v2`)
- Пакетная обработка с настраиваемым `RERANKER_BATCH_SIZE` (по умолчанию: 32)
- Настраиваемый `MAX_CHUNKS_AFTER_RERANK` (по умолчанию: 20)
- Поддержка удалённого реранкера с локальным фолбэком

#### Сборщик контекста (`context.py`)

- Дедупликация через SHA-256 хеширование содержимого
- Фильтрация с учётом версий (извлечение версии из запроса или параметра `rag_version`)
- Сборка с бюджетом токенов и настраиваемыми лимитами
- Обогащение метаданными для цитирования источников

#### LLM Роутер (`llm/provider.py`)

- Мульти-провайдерный адаптер: vLLM, llama.cpp, Anthropic, Ollama, OpenAI-совместимый
- Потоковая и непотоковая генерация
- Логика повторных попыток с экспоненциальной задержкой
- Провайдер-специфичная трансляция функций (поле `system` Anthropic, блок `options` Ollama)

#### Оценка уверенности (`confidence.py`)

- Эвристическая оценка на основе качества поиска и покрытия контекста
- Опциональная NLI-верификация (проверка entailment по контексту)
- CRAG-оценщик для оценки качества поиска
- Настраиваемые пороги для эскалации

#### Оптимизатор токенов (`token_optimizer.py`)

- BPE-подсчёт токенов (tiktoken)
- 4 стратегии сжатия: усечение, обогащение заголовков, расширение контекста, иерархическая
- Умное распределение бюджета между системным промптом, контекстом и ответом

### 4.3 Граф знаний

Граф знаний обогащает поиск связями между сущностями.

- **10 типов сущностей**: Person, Document, Project, Component, Technology, Team, Meeting, Decision, Milestone, Issue
- **9 типов связей**: authored_by, belongs_to, depends_on, related_to, mentions, created_by, assigned_to, reviewed_by,
  blocks
- **Многошаговый обход**: Cypher-запросы обходят связи для поиска связанных сущностей в пределах настраиваемой глубины
- **Graceful degradation**: Neo4j недоступен → расширение графа пропускается, прокси продолжает работать только с
  векторным поиском

### 4.4 Кэширование

Многоуровневое кэширование с настраиваемыми бэкендами:

| Уровень        | Бэкенд            | TTL   | Содержимое                     |
|----------------|-------------------|-------|--------------------------------|
| **Эмбеддинги** | Redis / in-memory | 24ч   | Эмбеддинги запросов (MD5 ключ) |
| **Реранк**     | Redis / in-memory | 5 мин | Индексы реранжированных чанков |
| **Ответ**      | Redis / in-memory | 1ч    | Полные ответы LLM              |

Когда Redis недоступен (`USE_REDIS=false` или сбой подключения), система переключается на LRU-кэш в памяти. Прокси
никогда не падает при сбое кэша.

## 6. Архитектура развёртывания

### Автономный дизайн

Система спроектирована для полностью офлайн-работы:

1. **Предзагруженные модели**: Все модели (эмбеддер, реранкер, LLM, SLM) загружаются через
   `scripts/download_models_offline.py` и хранятся локально.
2. **Нет внешних API-вызовов**: Весь инференс работает локально — без вызовов OpenAI, Cohere или HuggingFace API во
   время выполнения.
3. **Офлайн Docker-образы**: Все образы контейнеров предварительно загружены и доступны локально.
4. **Локальное обслуживание моделей**: vLLM или llama.cpp обслуживает модели на localhost.

### Docker Compose (одиночный сервер)

```yaml
# proxy/docker-compose.yml
services:
  qdrant:    # Векторная база данных (порт 6333)
  neo4j:     # Графовая база данных (порт 7687)
  redis:     # Кэш (порт 6379)
  vllm:      # LLM бэкенд (порт 8000)
  rag-proxy: # FastAPI прокси (порт 8080)
```

### Kubernetes (продакшен)

- **Helm-чарт**: `k8s/helm/rag-system/`
- **Horizontal Pod Autoscaler**: Масштабирует поды прокси на основе CPU/частоты запросов
- **Liveness/Readiness пробы**: `/v1/health/live` и `/v1/health/ready`
- **Управление секретами**: Kubernetes secrets для JWT_SECRET, API ключей, паролей
- **Сетевые политики**: Ограничение межсервисного взаимодействия
- **Персистентные тома**: Сохранение данных Qdrant, Neo4j, Redis

## 7. Модель безопасности

### Аутентификация

| Метод               | Статус        | Описание                                                      |
|---------------------|---------------|---------------------------------------------------------------|
| **JWT (локальный)** | Реализован    | Пары access + refresh токенов, SQLite хранилище пользователей |
| **LDAP/AD**         | Реализован    | Фолбэк аутентификации через корпоративный каталог             |
| **Keycloak OIDC**   | Настраиваемый | Интеграция корпоративного SSO                                 |

### Авторизация (RBAC)

| Роль        | Ранг | Разрешения                                          |
|-------------|------|-----------------------------------------------------|
| `admin`     | 4    | Все эндпоинты, управление моделями, запуск обучения |
| `expert`    | 3    | Чат + отправка обратной связи                       |
| `user`      | 2    | Только чат                                          |
| `read_only` | 1    | Список моделей + проверки здоровья                  |

### Функции безопасности

- **Санитизация ввода**: Защита от SQL-инъекций, XSS и ограничение длины
- **Ограничение частоты запросов**: Token bucket по IP (настраиваемые `RATE_LIMIT_PER_MINUTE`, `RATE_LIMIT_BURST`)
- **Защита от брутфорса**: Ограничение попыток входа (5 попыток за 5 мин, 15 мин кулдаун)
- **Маскирование секретов**: Чувствительные значения редактируются в логах
- **Контроль доступа на уровне строк**: Фильтрация ACL документов после поиска
- **Чёрный список токенов**: Отозванные access-токены заносятся в чёрный список
- **CORS**: Настраиваемые разрешённые источники
- **Сжатие**: gzip сжатие ответов

## 8. Ключевые принципы проектирования

1. **Автономность прежде всего** — все модели предзагружены, никаких внешних API-вызовов при выполнении.
2. **Graceful degradation** — каждый компонент может отказать независимо. Neo4j недоступен → пропуск расширения графа.
   Реранкер OOM → использование сырых оценок. Redis недоступен → кэш в памяти. Прокси никогда не падает.
3. **Инкрементальность по умолчанию** — WAL-чейкпоинтинг. SHA-256 контентно-адресуемые чанки. Переиндексируются только
   изменённые документы.
4. **OpenAI-совместимость** — полная замена любого OpenAI-клиента. RAG-расширения молча игнорируются стандартными
   клиентами.
5. **Двухмодельная маршрутизация** — SLM (2-3B) для быстрой предобработки. LLM для тяжёлой генерации.
6. **Мульти-провайдерность** — подключаемые адаптеры бэкендов для vLLM, llama.cpp, Anthropic, Ollama,
   OpenAI-совместимых.
7. **Опциональная сложность** — LangGraph, Neo4j, Redis все опциональны. Работает в простом RAG-режиме по умолчанию.
8. **Экономия токенов** — BPE-подсчёт, 4 стратегии сжатия, умное распределение бюджета.
