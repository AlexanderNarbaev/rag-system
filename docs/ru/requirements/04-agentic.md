# Блок D. Агентская оркестрация — LangGraph (FR-26 — FR-31)

---

## FR-26. LangGraph 10-node graph

**Описание:**
Система опционально использует LangGraph для агентской оркестрации. Граф из 10 нод:

1. `rewrite` — переформулировка запроса
2. `retrieve` — гибридный поиск
3. `check_sufficiency` — проверка достаточности контекста
4. `graph_expand` — расширение через граф знаний
5. `rerank` — ранжирование
6. `build_context` — сборка контекста
7. `generate` — генерация ответа LLM
8. `check_confidence` — проверка уверенности
9. `call_tools` — вызов инструментов (если LLM запросил)
10. `finalize` — финализация ответа

**Критерий приёмки:**

1. `USE_LANGGRAPH=true` — граф компилируется без ошибок
2. Запрос проходит через все ноды по порядку
3. Лог содержит записи для каждой ноды

**Статус:** ⚠️ Код есть (`proxy/app/core/orchestrator/graph.py`), нужен интеграционный тест
**Приоритет:** HIGH (opt-in)
**Связь:** ADR-006

## FR-27. Query rewriting (SLM/LLM)

**Описание:**
Перед поиском система переформулирует запрос пользователя для улучшения релевантности:

- Исправляет опечатки
- Расширяет аббревиатуры
- Добавляет контекст из предыдущих сообщений (для multi-turn)

**Критерий приёмки:**

1. Запрос с опечаткой — rewrite исправляет её
2. Rewrite-запрос отличается от оригинального
3. Результаты поиска по rewrite-запросу более релевантны

**Статус:** ⚠️ Код есть (`proxy/app/core/orchestrator/nodes.py`), нужен интеграционный тест
**Приоритет:** HIGH (opt-in)
**Связь:** ADR-006

## FR-28. Retrieval sufficiency loop (max 3)

**Описание:**
После поиска система проверяет достаточность контекста (score ≥ 0.6). Если
недостаточно — возвращает к ноде `rewrite` для новой попытки. Максимум 3 цикла.
После 3 неудачных попыток — используется лучший из найденных контекстов.

**Критерий приёмки:**

1. Недостаточный контекст — система переписывает запрос и ищет снова
2. Максимум 3 цикла (не бесконечный loop)
3. После 3 циклов — используется лучший найденный контекст

**Статус:** ⚠️ Код есть (`proxy/app/core/orchestrator/nodes.py`), нужен интеграционный тест
**Приоритет:** HIGH (opt-in)
**Связь:** ADR-006

## FR-29. Fallback к линейному пайплайну

**Описание:**
При `USE_LANGGRAPH=false` система использует простой линейный пайплайн:
search → rerank → context → generate. Без циклов, без агентского поведения.

**Критерий приёмки:**

1. `USE_LANGGRAPH=false` — запрос обрабатывается линейно
2. Ответ формируется успешно
3. В логе нет упоминания LangGraph

**Статус:** ⚠️ Код есть (`proxy/app/main.py`), нужен интеграционный тест
**Приоритет:** CRITICAL
**Связь:** ADR-006, ADR-011

## FR-30. Tool/function calling

**Описание:**
LLM может запрашивать вызов инструментов (tool calls). Система поддерживает:

- Встроенные инструменты (live Confluence, Jira, GitLab API)
- Пользовательские инструменты (через SDK или declarative YAML)
- OpenAPI auto-discovery (автоматическое создание инструментов из OpenAPI-специй)

**Критерий приёмки:**

1. LLM запрашивает tool call — система выполняет его и возвращает результат
2. Результат tool call передаётся обратно в LLM для генерации финального ответа
3. Инструменты фильтруются по роли пользователя

**Статус:** ⚠️ Код есть (`proxy/app/tools/`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** ADR-009

## FR-31. Параллельное выполнение инструментов

**Описание:**
Если LLM запрашивает несколько tool calls одновременно, система выполняет их
параллельно через `asyncio.gather`. Зависимости между инструментами разрешаются
через топологическую сортировку.

**Критерий приёмки:**

1. Два независимых tool call — выполняются параллельно (время ≈ max(a, b), не a+b)
2. Зависимые tool call — выполняются последовательно
3. Ошибка одного инструмента не прерывает другие

**Статус:** ⚠️ Код есть (`proxy/app/tools/orchestrator.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** ADR-009 4.2-4.3
