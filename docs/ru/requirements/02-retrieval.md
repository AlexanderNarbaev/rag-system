# Блок B. Гибридный поиск и ранжирование (FR-09 — FR-18)

---

## FR-09. Гибридный поиск — dense + sparse RRF

**Описание:**
Система выполняет гибридный поиск, комбинируя:

- **Dense** — векторный поиск по 1024-мерным эмбеддингам BGE-M3
- **Sparse** — лексический поиск по разреженным векторам (BM25-стиль, тоже из BGE-M3)
- **Fusion** — результаты объединяются через Reciprocal Rank Fusion (RRF) с k=60

Формула RRF: `score(d) = Σ 1/(k + rank_i(d))` где k=60, сумма по всем методам поиска.

**Критерий приёмки:**

1. Поисковый запрос возвращает результаты из обоих методов (dense и sparse)
2. RRF-скор каждого результата — сумма рангов из обоих методов
3. Итоговый список отсортирован по RRF-скору (убывание)
4. Функция `hybrid_search()` возвращает Qdrant ScoredPoint-ы с payload

**Статус:** ⚠️ Код есть (`proxy/app/core/retrieval.py`), нужен интеграционный тест
**Приоритет:** CRITICAL
**Связь:** ADR-001, ADR-002

## FR-10. Cross-encoder reranking

**Описание:**
После гибридного поиска top-N результатов (по умолчанию 50) система ранжирует их
заново с помощью cross-encoder модели BGE-Reranker-v2-m3. Cross-encoder получает
пару (запрос, документ) и выдаёт релевантность. Это точнее, чем bi-encoder, но
медленнее, поэтому применяется только к top-N.

Параметры: `batch_size=32`, модель BGE-Reranker-v2-m3 (поддержка 100+ языков).

**Критерий приёмки:**

1. После reranking топ-20 результатов порядок отличается от исходного (reranker переупорядочивает)
2. Результаты с низкой релевантностью отсекаются
3. Функция `rerank_chunks()` возвращает отсортированные индексы

**Статус:** ✅ Подтверждено (`proxy/app/core/rerank.py`)
**Приоритет:** CRITICAL
**Связь:** ADR-002

## FR-11. Дедупликация по содержимому (SHA-256)

**Описание:**
При сборке контекста система удаляет дубликаты чанков, используя SHA-256 хеш
содержимого. Если два чанка имеют одинаковый хеш — оставляется только один (с более
высоким скором). Это предотвращает повторение одного и того же текста в контексте.

**Критерий приёмки:**

1. Два идентичных чанка — в контексте остаётся только один
2. Два похожих (но не идентичных) чанка — оба остаются
3. Функция `deduplicate_chunks()` уменьшает количество чанков при наличии дубликатов

**Статус:** ⚠️ Код есть (`proxy/app/core/context/builder.py`), нужен интеграционный тест
**Приоритет:** CRITICAL
**Связь:** ADR-005

## FR-12. Version-aware фильтрация

**Описание:**
Каждый чанк в Qdrant имеет поле `version` (строка, например "v1", "v2"). Параметр
`rag_version` фильтрует результаты поиска, возвращая только чанки указанной версии.
Если версия не указана — возвращаются чанки последней версии. Фильтрация учитывает
не только версию, но и ACL-метаданные чанка (access_level, allowed_groups,
allowed_users).

**Критерий приёмки:**

1. `rag_version="v1"` — все возвращённые чанки имеют `version="v1"`
2. Без параметра — возвращаются чанки с любой версией (последняя приоритетнее)

**Статус:** ⚠️ Код есть (`proxy/app/core/retrieval.py`), нужен интеграционный тест
**Приоритет:** CRITICAL
**Связь:** ADR-005

## FR-13. Кэширование эмбеддингов

**Описание:**
Эмбеддинги пользовательских запросов кэшируются в двух уровнях:

- In-memory LRU-кэш (быстрый, теряется при перезапуске)
- Redis (опционально, персистентный)

Ключ кэша — MD5-хеш текста запроса. При повторном запросе эмбеддинг берётся из кэша
без вызова модели.

**Критерий приёмки:**

1. Повторный запрос того же текста — эмбеддинг берётся из кэша (лог: "Embedding cache hit")
2. Метрика `rag_cache_hit_ratio{cache_type="embedding"}` ≥ 60% при повторных запросах

**Статус:** ⚠️ Код есть (`proxy/app/core/retrieval.py`), нужен интеграционный тест
**Приоритет:** CRITICAL
**Связь:** ADR-001

## FR-14. ColBERT late-interaction retrieval

**Описание:**
BGE-M3 производит не только dense-эмбеддинги (1024-dim), но и ColBERT-вектора —
множественные вектора на токен. ColBERT позволяет более точно вычислять
релевантность через late interaction (сравнение токен-уровневых векторов).

Система поддерживает ColBERT как дополнительный канал поиска в гибридном режиме.

**Критерий приёмки:**

1. ColBERT-поиск возвращает результаты
2. Результаты комбинируются с dense/sparse через RRF

**Статус:** ⚠️ Код есть (`proxy/app/core/retrieval.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** roadmap Phase 1

## FR-15. Knee-point pruning

**Описание:**
После гибридного поиска система анализирует распределение скоров и отсекает
результаты ниже «knee point» — точки резкого падения релевантности. Это убирает
шумовые результаты, которые только раздувают контекст.

**Критерий приёмки:**

1. При наличии выраженного knee point — результаты ниже отсекаются
2. При равномерном распределении — все результаты сохраняются
3. Количество результатов после pruning ≤ количества до pruning

**Статус:** ⚠️ Код есть (`proxy/app/core/retrieval.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** roadmap Phase 2

## FR-16. FLARE — Forward-Looking Active Retrieval

**Описание:**
При генерации ответа система анализирует, генерирует ли LLM «неуверенный» токен
(вероятность < порога). Если да — система приостанавливает генерацию, формирует
новый поисковый запрос из уже сгенерированного текста, получает дополнительные
чанки и продолжает генерацию с расширенным контекстом.

**Критерий приёмки:**

1. При генерации с низкой уверенностью —запускается дополнительный поиск
2. Генерация продолжается с новыми чанками
3. Финальный ответ содержит информацию из дополнительных чанков

**Статус:** ⚠️ Код есть (`proxy/app/core/flare.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** roadmap Phase 5

## FR-17. Two-stage reranking

**Описание:**
Двухэтапное ранжирование:

1. Первый этап — быстрый bi-encoder (BGE-M3) для грубой фильтрации top-100 → top-50
2. Второй этап — точный cross-encoder (BGE-Reranker-v2-m3) для финального top-50 → top-20

Это снижает latency по сравнению с однопроходным cross-encoder на всех результатах.

**Критерий приёмки:**

1. nDCG@10 двухэтапного ранжирования ≥ nDCG@10 однопроходного
2. Latency двухэтапного < latency однопроходного (при top-100 входе)

**Статус:** ⚠️ Код есть (`proxy/app/core/rerank.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** roadmap Phase 5

## FR-18. Dynamic top-k на основе SLM

**Описание:**
SLM классифицирует сложность запроса (simple/medium/complex). На основе классификации
динамически выбирается количество чанков для поиска:

- simple → top_k=5 (меньше контекста, быстрее)
- medium → top_k=10
- complex → top_k=20

Это снижает latency для простых запросов и увеличивает качество для сложных.

**Критерий приёмки:**

1. Простой запрос (например, "какая дата?") — извлекается ≤ 5 чанков
2. Сложный запрос (например, "сравни архитектуру X и Y") — извлекается ≥ 15 чанков
3. Лог содержит "Query classified as 'simple'/'complex'"

**Статус:** ⚠️ Код есть (`proxy/app/core/query_router.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** roadmap Phase 3
