## FR-40. Извлечение из 6 источников данных

**Описание:**
ETL-пайплайн извлекает данные из:

1. **Confluence** — страницы, комментарии, вложения
2. **Jira** — тикеты, описания, комментарии
3. **GitLab** — README, wiki, issues, merge requests
4. **Documents** — PDF, DOCX, MD, TXT, HTML
5. **Books** — PDF-книги с OCR
6. **Chats** — логи чатов (Telegram, Slack)

**Критерий приёмки:**

1. Каждый экстрактор возвращает список документов с метаданными
2. Каждый документ имеет: id, title, content, source_type, metadata
3. Инкрементальное извлечение — только изменённые документы

**Статус:** ⚠️ Код есть (6 экстракторов в `etl/extractors/`), нужен интеграционный тест
**Приоритет:** CRITICAL

## FR-41. Semantic chunking

**Описание:**
Документы разбиваются на чанки с учётом семантики:

- По заголовкам (H1-H6)
- По абзацам с сохранением контекста
- С overlap (50-100 токенов между чанками)
- С сохранением структуры таблиц и кода

Каждый чанк включает ACL-метаданные: `access_level`
(public/internal/confidential/restricted), `allowed_groups`, `allowed_users`.
Значения по умолчанию: public/пустые списки. Реальные значения извлекаются из
метаданных источника (например, права доступа Confluence-пространства).

**Критерий приёмки:**

1. Чанки не разрывают предложения
2. Каждый чанк содержит контекст (заголовок секции, имя документа)
3. Overlap между соседними чанками — 50-100 токенов

**Статус:** ⚠️ Код есть (`etl/chunker/semantic_chunker.py`), нужен интеграционный тест
**Приоритет:** CRITICAL
**Связь:** AGENTS.md

## FR-44. WAL-based incremental extraction

**Описание:**
Write-Ahead Log отслеживает состояние каждого этапа ETL. При сбое ETL
возобновляется с последнего успешного этапа, а не с начала.

**Критерий приёмки:**

1. Сбой ETL на этапе embedding — повторный запуск начинается с embedding (не с extraction)
2. WAL-файл содержит checkpoint для каждого этапа
3. Успешное завершение — WAL очищается

**Статус:** ⚠️ Код есть (`etl/indexer/wal_manager.py`), нужен интеграционный тест
**Приоритет:** CRITICAL
**Связь:** ADR-005, ADR-011

## FR-45. SHA-256 content-addressable chunks

**Описание:**
Каждый чанк хешируется SHA-256 от содержимого. Хеш используется как:

- ID точки в Qdrant (Point ID)
- Ключ для дедупликации
- Ключ для версионирования (если содержимое изменился — новый хеш = новая версия)

**Критерий приёмки:**

1. Одинаковый контент → одинаковый хеш → одна точка в Qdrant
2. Изменённый контент → другой хеш → новая точка в Qdrant
3. Хеш используется как Point ID в Qdrant

**Статус:** ⚠️ Код есть (`etl/chunker/hash_versioning.py`), нужен интеграционный тест
**Приоритет:** CRITICAL
**Связь:** ADR-005

## FR-46. Hot/cold storage stratification

**Описание:**

- **Hot storage** — текущая версия документов в Qdrant (быстрый доступ)
- **Cold storage** — предыдущие версии в Parquet-файлах (дешёвое хранение)
- При запросе конкретной версии (`rag_version`) — поиск и в hot, и в cold

**Критерий приёмки:**

1. Текущая версия — в Qdrant (hot)
2. Старая версия — в Parquet (cold)
3. `rag_version="v1"` — находит чанки даже если v1 в cold storage

**Статус:** ⚠️ Код есть (`etl/indexer/live_vector_lake.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** ADR-005

## FR-47. Version tracking

**Описание:**
Каждый чанк имеет поле `version` (строка). При обновлении документа — старый чанк
помечается как устаревший, новый — с новой версией. Система отслеживает историю
версий.

**Критерий приёмки:**

1. Обновлённый документ — старые чанки получают `stale=true`, новые — `version=current`
2. Запрос `rag_version` — возвращает чанки только указанной версии
3. Без `rag_version` — возвращает чанки текущей версии

**Статус:** ⚠️ Код есть (`etl/chunker/hash_versioning.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** ADR-005

## FR-48. RAPTOR hierarchical indexing

**Описание:**
Иерархическая индексация: чанки кластеризуются, для каждого кластера генерируется
summary. Summary кластеров снова кластеризуются — и так до вершины дерева. При
поиске система может отвечать на разном уровне детализации.

**Критерий приёмки:**

1. Дерево summary построено (корень → кластер → чанки)
2. Поиск на верхнем уровне — возвращает общие summary
3. Поиск на нижнем уровне — возвращает конкретные чанки

**Статус:** ⚠️ Код есть (`etl/indexer/tree_builder.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** roadmap Phase 2

## FR-49. Code-aware chunking (AST-based)

**Описание:**
Исходный код разбивается на чанки по AST-структуре:

- Python — по функциям и классам
- JavaScript/TypeScript — по функциям и классам
- Java — по методам и классам

Каждый чанк сохраняет контекст (имя файла, имя функции/класса, импорты).

**Критерий приёмки:**

1. Python-файл с 3 функциями — 3 чанка (по одному на функцию)
2. Каждый чанк содержит имя функции в метаданных
3. Импорты дублируются в каждом чанке (для контекста)

**Статус:** ⚠️ Код есть (`etl/chunker/code_chunker.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** roadmap Phase 5.2

## FR-51. Quality metrics для чанков

**Описание:**
Система вычисляет метрики качества для каждого чанка:

- Semantic coherence — косинусное сходство предложений внутри чанка
- Information density — количество уникальных терминов
- Completeness — содержит ли чанк полную мысль

**Критерий приёмки:**

1. Каждый чанк имеет `quality_score` (0-1)
2. Чанки с quality_score < порога — фильтруются или обогащаются
3. Метрики логируются

**Статус:** ⚠️ Код есть (`etl/indexer/chunk_quality.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** quality_metrics.py

## FR-52. Chunk enrichment (SLM)

**Описание:**
SLM обогащает чанки метаданными:

- Генерирует summary чанка
- Извлекает ключевые сущности
- Определяет тематику

Обогащённые метаданные хранятся в payload чанка в Qdrant.

**Критерий приёмки:**

1. Каждый чанк имеет `summary` (сгенерированный SLM)
2. Каждый чанк имеет `entities` (извлечённые SLM)
3. Обогащение не ломает существующие метаданные

**Статус:** ⚠️ Код есть (`etl/indexer/chunk_enricher.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** chunk_enricher.py

## FR-53. Streaming pipeline

**Описание:**
ETL работает в streaming-режиме: документы обрабатываются по мере поступления
(а не пакетами). Webhook-сервер принимает уведомления от Confluence/Jira/GitLab
и запускает обработку немедленно.

**Критерий приёмки:**

1. Webhook-событие → документ обрабатывается в течение 5 секунд
2. Обработанный чанк доступен для поиска немедленно
3. Ошибка обработки — retry с exponential backoff

**Статус:** ⚠️ Код есть (`etl/scheduler/streaming_pipeline.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** NFR-P10

## FR-54. Event pipeline

**Описание:**
Система обрабатывает события из внешних источников:

- Confluence: page_created, page_updated, page_removed
- Jira: issue_created, issue_updated, comment_added
- GitLab: push, merge_request, wiki_updated

Каждое событие запускает соответствующий ETL-процесс.

**Критерий приёмки:**

1. Событие page_updated → документ переиндексируется
2. Событие page_removed → чанки удаляются из Qdrant
3. Неизвестное событие — логируется, не падает

**Статус:** ⚠️ Код есть (`etl/scheduler/event_pipeline.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** event_pipeline.py

## FR-55. Webhook server

**Описание:**
HTTP-сервер принимает webhook-уведомления от внешних систем. Эндпоинты:

- `POST /webhook/confluence` — события Confluence
- `POST /webhook/jira` — события Jira
- `POST /webhook/gitlab` — события GitLab

**Критерий приёмки:**

1. POST-запрос с валидным payload — обрабатывается успешно
2. POST-запрос с невалидным payload — 400 Bad Request
3. Webhook-секрет проверяется (HMAC signature)

**Статус:** ⚠️ Код есть (`etl/scheduler/webhook_server.py`), нужен интеграционный тест
**Приоритет:** HIGH
**Связь:** webhook_server.py

## FR-56. Task scheduler

- Полная индексация — раз в сутки

## FR-57. Cold storage cleanup

**Описание:**
Задача по расписанию очищает cold storage:

- Удаляет версии старше 90 дней
- Архивирует версии старше 30 дней в S3/MinIO
- Логирует количество удалённых/архивированных записей

**Критерий приёмки:**

1. Версии > 90 дней — удаляются
2. Версии 30-90 дней — архивируются в S3
3. Лог содержит количество обработанных записей

**Статус:** ⚠️ Код есть (`etl/scheduler/cold_storage_cleanup.py`), нужен интеграционный тест
**Приоритет:** MEDIUM
**Связь:** performance-quality 6.4
